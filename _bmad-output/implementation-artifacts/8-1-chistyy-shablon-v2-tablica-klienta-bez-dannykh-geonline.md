## Пользовательская история

Как **коуч практики (Азиза)**,
я хочу **чтобы созданная ботом таблица нового клиента содержала только его данные**,
чтобы **пилот с реальным клиентом был возможен и данные Geonline не утекали третьим лицам**.

## Контекст и границы scope

Прод-баг (719549e, HIGH, блокер пилота): `F0_SHEETS_TEMPLATE_ID` указывает на копию **боевой Geonline-таблицы**. `files.copy` переносит все 23+ листа с данными Geonline (13 скрытых legacy + видимые панели с именами топов Geonline), а F0 перезаписывает только машинные листы. Клиент AIPLUS открыл таблицу и увидел OKR-панели Geonline. Решение — двойное: **Fix A** (чистый шаблон без чужих данных) + **Fix B** (панели — live-формулы поверх машинных листов, рендерятся под любого клиента без кода).

**В scope:**
- **Генератор шаблона `scripts/f0-build-template.ts`** (шаблон-как-код, §1 дизайна): строит эталонный spreadsheet с нуля через Sheets API (OAuth-креды из 7.4). Состав листов — §3: видимые панели «📊 Все OKR», «🧪 Банк гипотез», «📅 Лог встреч», «📋 Ограничения (узелки)», «📖 Методология» + скрытый эталон «👤 Шаблон топа»; ⚙️-данные `_meta` (новый), `_okr`, `_stakeholder_map`, `_hypotheses`, `_f5_metrics`, `_ops_logs` — видимые, оформленные как рабочие (тёмная шапка, frozen row, фильтры). Формулы RU-локали (`;`, §5), панели под warning-протекцией.
- **Расширение схем записи** в `src/f0-sheets.ts` (§3): запись `_meta` (company/period/onboarding_date/tracker); `_stakeholder_map` + колонка `telegram` (контакт из диалога дозаполнения, сейчас уезжает только в card.json); `_hypotheses` + `okr_link`/`owner`/`deadline`/`status` (null где F0 не извлёк). Запись остаётся header-driven — колонки по именам заголовков.
- **Персональные листы топов** (§4): после записи данных — для каждого уникального `owner` из `_okr` `batchUpdate({duplicateSheet})` эталона → rename «👤 {Имя}» → `$B$1` = имя → показать; эталон скрыт. Идемпотентно: лист с таким именем есть → пропустить (retry `/confirm` не плодит листы).
- **Чистка прода** (ручное, Manual TODO): переключить `F0_SHEETS_TEMPLATE_ID`, удалить orphan AIPLUS, пере-онбордить AIPLUS.

**Вне scope:** legacy-листы Geonline — не переносятся вовсе (§6); Apps Script — отклонён (§1); правка боевой Geonline-таблицы и `GEONLINE_F0_SHEET_ID`; формульная переработка всех панелей сверх перечисленных; изменения `readClientContext` (F1 читает те же листы; новые колонки безопасны — чтение тоже header-driven, проверить регрессом).

**Известное ограничение MVP (утверждено, §6):** повторный `/confirm` поверх «пожившей» таблицы затирает правки трекера в `_okr` (`batchClear` + перезапись). Не чиним в 8.1; опция на будущее — предупреждение при confirm, если таблица редактировалась после онбординга.

## Acceptance Criteria

1. **Given** чистый бланк-шаблон v2.0, сгенерированный `scripts/f0-build-template.ts` (без данных и имён Geonline, вкл. legacy-листы), и `F0_SHEETS_TEMPLATE_ID`, указывающий на него, **when** онбординг создаёт таблицу клиента, **then** ни один лист созданной таблицы не содержит данных/имён Geonline; машинные листы `_meta`/`_okr`/`_stakeholder_map`/`_hypotheses` заполнены данными клиента (Fix A; схемы расширены по §3 дизайна).
2. **Given** формульные панели шаблона («📊 Все OKR», «🧪 Банк гипотез», эталон «👤 Шаблон топа») и запись F0 в машинные листы, **when** таблица создана, **then** панели отображают данные онбордингового клиента без участия кода (Fix B), а персональные листы «👤 {Имя}» созданы duplicateSheet-ом по каждому владельцу KR (идемпотентно — повторный `/confirm` не плодит листы).
3. **Given** orphan-таблица AIPLUS (`1JwqnA0T1GXD0_LvmrvYcf93z70C7Mkk8r089DHwwkjA`) и переключённый шаблон, **when** чистка и пере-онбординг AIPLUS выполнены, **then** orphan удалена, новая таблица AIPLUS корректна, а боевая таблица Geonline и `GEONLINE_F0_SHEET_ID` не затронуты (регресс-guardrail зелёный).

## Задачи

1. **`scripts/f0-build-template.ts`** — генератор шаблона: создание spreadsheet, листы/заголовки/формулы/легенда/форматирование по §3–§5, warning-протекции панелей, скрытие «👤 Шаблон топа», порядок листов (панели → лог → данные). Повторный запуск на существующем fileId — пересборка (идемпотентная перегенерация структуры).
2. **Схемы записи `f0-sheets.ts`**: маппер `_meta`; `mapStakeholderRows` + `telegram`; `mapHypothesisRows` + 4 колонки (null-заполнение); типы синхронно. Юнит-тесты мапперов и alignment по новым заголовкам.
3. **`duplicateSheet` персональных листов** в `createClientSpreadsheet`: уникальные owners из `_okr`, rename + `$B$1` + показать, скрыть эталон; идемпотентность по имени листа; ошибки → `F0SheetsError` с `spreadsheetId`. Юнит-тесты (моки batchUpdate: создание, пропуск существующих, сохранение retry-семантики 7.4).
4. **Смоук**: расширить `scripts/f0-sheets-smoke.ts` под новый шаблон — LIVE копия, запись расширенных схем, проверка что панели/персональные листы отрисовали данные тест-клиента и нигде нет строки «Geonline»/имён Geonline-топов.
5. **Регресс-guardrail**: canary + golden + весь vitest + tsc зелёные; путь `clientId === 'geonline'` / `GEONLINE_F0_SHEET_ID` не изменён (боевая таблица не трогается вообще).
6. **Manual TODO (Тимур)** — см. Definition of Done.

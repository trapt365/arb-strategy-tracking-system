## Пользовательская история

Как **коуч практики (Азиза)**,
я хочу **отправить боту OKR-документ клиента и получить черновик панели OKR**,
чтобы **черновик рабочей среды появлялся за ≤ 15 минут без участия архитектора**.

## Контекст и границы scope

Первый вертикальный срез F0: команда «новый клиент» → приём **одного md/txt-файла** → извлечение через Claude → черновик панели OKR в Telegram с 🔴-пометками (инвариант 1) и блоком «Не распознано» (инвариант 3).

**В scope:** команда `/newclient` (+ текст «новый клиент»), per-chat сессия онбординга (in-memory, по паттерну `pendingEdits`), приём `message:document` (md/txt, лимит 20 MB Bot API), препроцессинг текста (мусорные vscode-remote URL из оглавлений), новый пайплайн `src/f0-onboarding.ts`, промпт `prompts/f0-okr-extraction.md`, черновик в чат, persist черновика в `data/.onboarding/` (warn-only).

**Вне scope (следующие stories):** docx/pdf и несколько файлов (7.2), гипотезы и участники (7.2), диалог дозаполнения и восстановление сессии после рестарта (7.3), создание Sheets (7.4), карточка клиента (7.5), мультиклиентность (7.6). client_id на этом срезе не назначается — черновик привязан к chatId+draftId.

**Эталон входа:** `C:\Users\Timur\Documents\Vault1\GeOnline OKR Framework 2026.md` — таблицы с rowspan-паттерном (Objective только в первой строке группы), RU/KZ code-switching в KR, даты «До 30.07.2026»/«Постоянно»/«Октябрь-Январь», ответственные одним именем, у большинства KR нет базы «с X до Y» (это и должен ловить инвариант 1; переписанные версии с базами лежат в колонке «Как улучшить» — их НЕ подставлять автоматически, инвариант 3).

## Acceptance Criteria

1. **Given** трекер запустил «новый клиент» и приложил md-файл OKR-документа (эталон GeOnline OKR Framework 2026), **when** пайплайн отработал, **then** в чат приходит черновик панели OKR (objective → KR: формулировка / база / цель / ответственный / срок) за ≤ 15 мин.
2. **Given** KR без числовой базы «с X до Y» или без ответственного, **when** черновик формируется, **then** KR помечен 🔴 с причиной и попадает в блок «требует дозаполнения» (инвариант 1).
3. **Given** фрагменты, которые не удалось надёжно извлечь, **when** черновик формируется, **then** они перечислены в «Не распознано» без выдуманных значений (инвариант 3); не-OKR-файл → понятная ошибка, сессия не ломается.

## Задачи

1. **Схема + валидатор инварианта 1.** `types.ts`: `F0OkrDraftSchema` (objectives → krs: formulation/base/target/owner/deadline, unrecognized[]); чистая функция `markBlockingKrIssues()` (нет числовой базы → 🔴 `no_base`, нет ответственного → 🔴 `no_owner`); unit-тесты на счётные/несчётные KR (образцы из эталона).
2. **Промпт `prompts/f0-okr-extraction.md`** ({{documentText}}): правила «не выдумывать» (незаполненное → null, сомнительное → unrecognized), rowspan-паттерн таблиц, RU/KZ, разноформатные даты как есть, «Как улучшить» не подставлять; few-shot фрагмент в `prompts/examples/`; запись в `prompts/CHANGELOG.md`.
3. **Препроцессор входа** `src/utils/f0-input.ts`: markdown/plain из файла, отсечение vscode-remote/file-URL мусора, guard размера/типа; unit-тесты.
4. **Бот-слой** (`bot.ts`): `/newclient` + «новый клиент» (только whitelist-чаты), сессия `Map<chatId, F0Session>`, хендлер `message:document` (getFile → download → препроцессор), прогресс-сообщения по паттерну `/report`, рендер черновика (🔴-блок «требует дозаполнения», блок «Не распознано») через `telegram-formatter`; ошибки → понятное сообщение, сессия остаётся живой.
5. **Пайплайн `src/f0-onboarding.ts`**: `runF0OkrDraft()` = loadPrompt → `callClaude` (schema, retry) → валидатор → структура черновика; persist `data/.onboarding/{draftId}.json` (warn-only); `scripts/f0-smoke.ts` на эталонном md; полный `npm test` + typecheck зелёные.

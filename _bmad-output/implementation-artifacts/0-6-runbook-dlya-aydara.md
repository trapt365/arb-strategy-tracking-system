# Story 0.6: Runbook для Айдара

Status: ready-for-dev (2026-04-22)

## Story (Пользовательская история)

Как **контроллер качества коучинга (Айдар, руководитель ARB)**,
Я хочу **иметь одностраничный runbook для экстренных ситуаций с AI-pipeline**,
Чтобы **если Тимур недоступен, я мог перезапустить систему, проверить статус и переключить Азизу в ручной режим работы, не теряя рабочий цикл клиента**.

## Контекст

Тимур — единственная точка ops (SPOF по ADR-архитектуре: риск #5 в [architecture.md](../planning-artifacts/architecture.md), таблица "Pre-mortem risks"). При отпуске/болезни/командировке у Айдара должна быть инструкция, покрывающая три типовых сбоя:

1. **Docker-контейнер упал** (нужен рестарт, возможно инспекция логов).
2. **AI-pipeline молчит > 30 мин** (Telegram отвечает, но отчёт не приходит).
3. **Telegram API / bot недоступен** (канал доставки сломан).

Версионирование runbook: **v1.0** — на текущей инфраструктуре (локальный Docker, без VPS). **v1.1+** — после Story 1.14 (разворачивание на Hostinger VPS) — добавить реальные креды, URL dashboard, точный SSH-доступ.

## Критерии приёмки

1. **Сценарий: Docker-контейнер упал**
   ```
   Дано Айдар получил алерт в ops-канале Telegram или Азиза сообщила что бот не отвечает
   Когда Айдар открывает runbook
   Тогда описана последовательность команд:
     - как подключиться (SSH на VPS / локальная машина Тимура)
     - как проверить статус: `docker compose ps` (ожидаемо: `healthy`)
     - как перезапустить: `docker compose restart app`
     - как посмотреть последние логи: `docker compose logs --tail=100 app`
     И указано что делать если рестарт не помог: эскалация Тимуру или Азизе переключение в ручной режим
   ```

2. **Сценарий: AI-pipeline не отвечает > 30 мин**
   ```
   Дано Азиза сообщила "отчёт не пришёл"
   Когда Айдар открывает runbook
   Тогда описан диагностический чеклист:
     - `/health` endpoint: `curl http://<host>:3000/health` → ожидаемо 200 `{"status":"ok"}`
     - проверить лимиты API: логи содержат "circuit breaker open" или "429 rate limit" (grep)
     - проверить внешний статус: status.anthropic.com, status.soniox.com (ссылки в runbook)
     И описан fallback: Азиза переводится в ручной режим (пишет отчёт из транскрипта без AI)
   ```

3. **Сценарий: Telegram-бот недоступен > 1 часа**
   ```
   Дано бот не отвечает в рабочем чате и в ops-канале
   Когда Айдар подтвердил что инфра жива (Docker healthy, /health 200)
   Тогда описан email-emergency mode:
     - Азиза пишет отчёт по шаблону (копия из aziza-runbook секции "Полный сбой")
     - Отправляет клиенту через email
     - CC на Айдара и Тимура
     - Фиксирует инцидент в Google Sheets лист `_ops_incidents` (дата, длительность, причина)
     И указан порог эскалации Тимуру: > 1 часа даунтайма в рабочие часы
   ```

4. **Сценарий: Айдар имеет доступ к инфраструктуре**
   ```
   Дано проект развёрнут (v1.0 — локально / v1.1 — на Hostinger VPS)
   Когда Айдар должен получить доступ
   Тогда креды хранятся в password manager (Bitwarden/1Password) с shared vault "ARB-ops"
     И runbook ссылается на vault, а не содержит пароли inline
     И проверено: Айдар может открыть vault и подключиться хотя бы раз (сухой прогон)
   ```

5. **Сценарий: Runbook помещается на 1 печатной странице A4**
   ```
   Дано Айдар распечатал runbook на A4 (или открыл PDF с телефона)
   Когда он ищет конкретное действие
   Тогда весь decision tree виден без скролла:
     - «Что сломалось?» → 3 ветви
     - Каждая ветвь: 3-5 команд или шагов
     - Контакты эскалации в футере
     И шрифт читаем на экране телефона (не мельче 10pt при печати)
   ```

## Заметки для разработчика

### Формат и стиль

- **Стилевой ориентир:** [docs/aziza-runbook-v1.0.md](../../docs/aziza-runbook-v1.0.md) — тот же тон, структура "что сломалось → что делать", footer с контактами.
- **Язык:** русский (первичный). Казахский перевод — не обязателен (Айдар свободно читает русский).
- **Объём:** **жёсткое ограничение 1 страница A4** (примерно 300-400 слов). Decision tree должен быть визуально сканируемым — использовать буллеты, bold для команд, не писать длинные абзацы.
- **Печатаемость:** PDF экспорт обязателен (Айдар распечатывает и кладёт на стол / в ящик).

### Команды и конфигурация — сверяться с актуальным кодом

Все команды `docker compose ...` должны соответствовать реальному [docker-compose.yml](../../docker-compose.yml) из Story 1.1 — сервис называется **`app`**, контейнер `tracking-app`, healthcheck по `http://localhost:3000/health`. Порт из env `PORT` (default 3000), volume `./data:/app/data:rw`.

`/health` endpoint фактический формат (из [src/server.ts](../../src/server.ts)):
```json
{"status":"ok","uptimeSeconds":123,"env":"production"}
```

Поддерживается `GET` и `HEAD` (фикс от code review Story 1.1) — можно использовать `curl -I` для быстрой проверки без тела.

### Что НЕ делать в runbook

- **Не инлайнить пароли/ключи** — всё через password manager. Если в runbook есть IP/host VPS — это OK, но никаких `sk-ant-*` или SSH private keys в Markdown.
- **Не писать "debug"-инструкции** — runbook для ситуации "надо быстро вернуть в строй", не для диагностики root cause. Root cause — это задача Тимура, когда он вернётся. Айдар восстанавливает работу, фиксирует инцидент, передаёт эстафету.
- **Не расширять scope** — если обнаружится что «надо бы добавить ещё 5 сценариев», фиксируем в `_ops_incidents` как feature request, в runbook v1.0 не вносим. Один лист, три ветви.

### Принципы эскалации

Runbook рассчитан на три уровня:

1. **Azize может сама:** мягкий fallback (повтор команды, ручной отчёт из транскрипта) — покрыто в aziza-runbook.
2. **Айдар может сам:** `docker compose restart`, проверить `/health`, переключить Азизу на ручной режим, активировать email-emergency mode. Это — данный runbook.
3. **Только Тимур:** новые конфигурации, миграции, API-ключи, код-изменения, провайдер-смены. Телефонный звонок Тимуру — нормальная процедура при `>1ч` даунтайма в рабочие часы.

### Почему Bitwarden/1Password, а не .env + SSH ключ в почте

ADR (неявный): credentials shared через manager — чтобы отзыв доступа был one-click (уволили/заболел — revoke access моментально). В `.env` / почте отозвать нельзя. Айдар получает `read-only` — он не может случайно сменить API-ключ и сломать прод. Это также пререквизит для будущего второго трекера (Марат в Epic 6).

### Почему v1.0 можно писать сейчас (до Story 1.14)

Скелет runbook не зависит от конкретного хоста — команды `docker compose ...` идентичны локально и на VPS. Что добавится в v1.1 после 1.14:
- реальный `ssh user@host` вместо placeholder,
- URL Hostinger dashboard,
- формат алертов из ops-канала (после Story 1.9 — ops logging & alerts),
- ссылка на ops-status бота (после Story 1.12).

До этого момента v1.0 — валидный артефакт для фазы «локальные тесты + первая живая встреча на машине Тимура с Telegram-доставкой». Его наличие — предпосылка Go/No-Go для production.

### Тест-стратегия

- **Чтение-ревью:** Айдар читает v1.0 → обратная связь (тест задачи 7.1).
- **Симуляция сбоя:** Тимур останавливает локальный контейнер (`docker compose stop`) → Айдар проходит runbook → поднимает обратно. Секундомер: восстановление < 5 мин.
- **Не требуется:** unit-тесты (это документ, не код), golden dataset, linting.

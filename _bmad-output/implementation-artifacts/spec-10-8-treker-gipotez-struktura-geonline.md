# Story 10.8 — Трекер гипотез: структура уровня geonline (обогащение F5)

## Проблема (почему нужна история)

10-5 построил F5 как **плоский отчёт** (список изменений статусов + счётчик + 3-5 выводов ИИ) — по своей спеке, где стоял «компактный Телеграм ≤4000 символов». Golden-eval против эталона Азизы (`geonline трекер гипотез неделя 12.md`, 10.07) показал структурный разрыв: эталон = богатый документ, разбитый по департаментам, с двумя таблицами на департамент, легендой, комментариями, ID, сводной матрицей. F5 не воспроизводит ничего из этого.

**Цель 10-8:** F5 выдаёт трекер, структурно совпадающий с эталоном Азизы. Плоская MVP-версия 10-5 заменяется на структурную. `computeDelta` и снимок недели переиспользуются.

## Целевая структура (ЖЁСТКИЙ критерий приёмки — сверять с reference_gold)

Отчёт ДОЛЖЕН содержать в этом порядке:

1. **Шапка** (таблица 2 колонки): Период (недели 1–N) · Подготовил (трекер) · Для (CEO/имя из профиля) · База (снимок прошлой недели: сколько гипотез, сколько департаментов).
2. **Легенда статусов** (таблица): 🟢 Работает · 🟡 В тесте · 🔴 Не работает · ⏳ Запланирована · ⛔/⏸️ Остановлена · 🆕 Новая · 📌 Без изменений.
3. **По каждому департаменту (OKR)** — секция `## N. НАЗВАНИЕ (OKR-X)`:
   - Строка **Ответственный:** имя · **Метрики:** ключевые цифры департамента.
   - Таблица **«Обновления статусов (нед.N-1 → нед.N)»**: колонки `# | Гипотеза | Статус нед.N-1 | Статус нед.N | Комментарий`. Комментарий содержит стрелку динамики (↑→↓), при наличии — прямую речь/цифру из встречи.
   - Таблица **«🆕 Новые гипотезы нед.N»**: колонки `# | Гипотеза | Статус | Запуск | Результат / Метрика | Следующий шаг`.
   - Департаменты без изменений и без новых — не выводить (или строкой «без обновлений»).
4. **Сводная матрица** (таблица): `Департамент | 🟢 | 🟡 | 🔴 | ⏳/🆕 | ⛔/⏸ | Всего | Δ нед.N-1→N`.
5. **Ключевые выводы** (буллеты, из LLM): динамика · подтверждённые · новые сильные · тревожные сигналы · рекомендация.
6. **Методологическая сноска**: какие встречи вошли, транскрипты (Soniox), версия листа/снимка.

ID гипотез (П-21, М-4, …) — брать из листа, если колонка есть; иначе генерировать префикс-по-департаменту + порядковый (`prefix(department)-index`).

## Источник данных для богатых ячеек (синтез, не ручное заполнение)

Комментарий / запуск / результат / следующий-шаг / прямая речь — НЕ требовать от трекера в листе. Синтезировать LLM из **отчётов по встречам недели** (F1, уже сохраняются per-клиент/неделя — тот же источник, что агрегирует недельный отчёт 9.7). То есть F5-rich = синтез над (лист `_hypotheses` для статусов+дельты + F1-отчёты недели для комментариев), сгруппированный по департаментам. Департамент/ответственный/метрики — из OKR-листа/профиля (как в 9.7). Это соответствует развороту («бот строит отчёты из гугл-шитс + аудио встреч»), не перекладывает работу на трекера.

Если F1-отчётов за неделю нет (встреч не было) → комментарии пустые, но структура (шапка/легенда/департаменты/статусы/сводка) всё равно строится из листа + дельты. Не блокировать.

## Доставка (rich не влезает в сообщение Телеграма)

Полный трекер (Markdown с таблицами) → отправить как **вложение-файл** (`.md`) в Телеграм (`sendDocument`), плюс **компактная сводка в чат**: заголовок + сводная матрица одной строкой на департамент + топ-3 сигнала + «полный трекер — во вложении». Паттерн — как 8.3/9.7 (компактно + полное отдельно), но носитель = файл, не Google-ссылка (не заводить Google Docs API). НЕ обрезать полный трекер через splitForTelegram.

## Boundaries & Constraints

- Клиентская изоляция: только активный клиент; `clientId === 'geonline'` fallback и `GEONLINE_F0_SHEET_ID` не трогать. Инварианты Epic 7 не ломать.
- Переиспользовать `computeDelta`, снимок `hypo-snapshot.json`, `readHypothesesSheet`. Снимок расширить: хранить не только статус, но и статус для дельты по департаментам (обратная совместимость: старый снимок = firstRun по новым полям, без падения — optional-поля zod).
- LLM-синтез комментариев: при ошибке Claude → строить структуру без комментариев (не partial, не блокировать), лог warn.
- Legacy: старый плоский `formatHypoReport` не удалять — переименовать в `formatHypoReportFlat` (может пригодиться как fallback), новый `formatHypoReportStructured` — основной. Кнопка «🧪 Трекер гипотез» → структурный.

## Tasks & Acceptance

1. **Данные департаментов + ID.** Расширить чтение: `_hypotheses` лист → колонки `department`, `okrLink`, `id` (если есть). OKR-лист/профиль → `responsible`, `metrics` per департамент (переиспользовать чтение из 9.7, если возможно). Группировка `HypoSnapshotItem[]` по департаменту.
2. **Синтез богатых ячеек (LLM).** Новый промпт `prompts/hypo-tracker-structured.md`: вход = статусы+дельта per департамент + F1-отчёты недели (текст); выход (JSON) = per-гипотеза `{id, comment, launch, result, nextStep}` + топ-выводы + метрики департаментов. Схема zod + `callClaude` + `withRetry` (как текущий conclusions-вызов).
3. **Структурный форматтер.** `formatHypoReportStructured(...)` → Markdown по «Целевой структуре» выше (шапка, легенда, per-департамент 2 таблицы, сводная матрица, выводы, сноска). Детерминированный скелет; LLM-ячейки вставляются.
4. **Доставка файлом.** bot.ts: обработчик `hypo_tracker:` → `runHypoTracker` возвращает `{compact: string, full: string}`; отправить `full` через `sendDocument` (.md) + `compact` в чат.
5. **Тесты (`f5-hypo-tracker.test.ts`):** структура форматтера (есть секции департаментов, обе таблицы, легенда, сводная матрица, порядок), группировка по департаментам, дельта per департамент, firstRun (нет снимка) строит полный список, пустой лист → сообщение, LLM-fail → структура без комментариев. Тесты судят СТРУКТУРУ (наличие/порядок секций и колонок), не читаемость.

**Acceptance Criteria:**
1. На тестовом входе (лист гипотез с ≥2 департаментами + ≥1 F1-отчёт) отчёт содержит: шапку, легенду, ≥2 секции департаментов, у каждого — «Обновления» и «Новые» таблицы с точными колонками из «Целевой структуры», сводную матрицу, выводы, сноску. Порядок секций совпадает.
2. Полный трекер уходит вложением-файлом; в чат — компактная сводка + топ-3 сигнала.
3. `computeDelta`/снимок переиспользованы; старый снимок не роняет пайплайн.
4. Gate зелёный: `tsc` + `vitest` + canary. Инварианты Epic 7 целы.

## Golden-eval (после реализации — делает главный агент, не движок)

Сгенерировать трекер на входе geonline нед.12 (лист-снимок нед.11 + F1/транскрипты нед.12) и сверить со структурой `reference_gold`. Критерий — структурное совпадение (департаменты, 2 таблицы, легенда, сводка, ID). Читаемость и точность комментариев — глазами пилота.

## Design Notes

- Эталон структуры — `reference_gold` (8 департаментов, ID П-*/М-*, 2 таблицы, комментарии с ↑↓ и прямой речью). НЕ копировать данные geonline в тесты — использовать синтетические фикстуры с ≥2 департаментами.
- Это open-loop история: unit-тесты гарантируют СКЕЛЕТ, но финальную приёмку качества даёт golden-eval глазами. Не считать «зелёные тесты» достаточным.

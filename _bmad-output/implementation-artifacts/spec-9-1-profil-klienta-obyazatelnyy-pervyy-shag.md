## Intent

**Problem:** `/newclient` стартует сразу со сбора документов/выбора способа — субъекты трекинга и фактура компании не зафиксированы до работы со стратегией, поэтому имена/названия в артефактах искажаются ASR или выдумываются LLM (аудио-фидбэк пилота 2026-07-09, CR-3).

**Approach:** Первым шагом онбординга — обязательный диалог «Профиль клиента» по Части A `docs/onboarding-questionnaire-v1.0.md` в два уровня: 🔑-минимум (A1.1 название, A1.2 суть, A3.2 топы: имя/должность/полномочия/зона, A3.3 decision maker) обязателен — до него способ онбординга стратегии не предлагается; расширенный (A1.3–A1.4, A2, A3.1, A4) опционален после минимума и доступен позже из карточки клиента. Данные — в расширенную `ClientCard` и в персист сессии (новая фаза `profile`).

## Boundaries & Constraints

**Always:**
- Все новые zod-поля — только `.optional()`/`.nullable()` (паттерн 8.5/8.6): старые `session-*.json` и `card.json` читаются без миграции.
- Persist после каждого ответа (механика 7.3); рестарт бота восстанавливает диалог профиля с текущего вопроса.
- Формулировки, примеры, порядок и 🔑-маркировка вопросов — дословно по Части A вопросника; по одному вопросу, заголовок блока при входе в блок, прогресс `(i/N)` отдельно для минимума и расширенной части.
- Числовые ответы (A2.1, A2.2, A2.5) — мягкая валидация 8.6: один переспрос, повтор принимается.
- Инвариант 3 (не выдумывать): неотвеченные поля остаются null/отсутствуют.
- `clientId === 'geonline'` fallback и `GEONLINE_F0_SHEET_ID` не трогать.

**Block If:**
- Требуется миграция формата `session-*.json` или `card.json` (несовместимое изменение).
- Требуется менять боевую таблицу Geonline или F1-цикл.

**Never:**
- Голосовые ответы (9.5), grounding в промптах (9.2), экран трёх способов и замена `F0_START_TEXT` (9.4) — вне scope; после профиля запускается СУЩЕСТВУЮЩИЙ flow сбора (F0_START_TEXT + f0ModeKeyboard) без изменений.
- Не смешивать вопросы профиля с очередью дозаполнения `filling` (F0Gap) — отдельная очередь.
- LLM в диалоге профиля не используется; содержимое файла A3.1 не обрабатывается.

## Tasks & Acceptance

**Execution:**
- `src/types.ts` -- добавить `ClientTopSchema` ({name, title, authority, area} — все кроме name nullable) и `ClientProfileSchema` (businessSummary, history, owners, financials {start: выручка/прибыльность/юнит-экономика/долги/…, target}, headcount, orgStructure, request (A4), tops, decisionMaker — всё optional); в `ClientCardSchema` — `profile: ClientProfileSchema.optional()`; в `F0PersistedSessionSchema` — `'profile'` в enum `phase` + optional `profile`, `profileQIndex` (и retry-индекс по образцу 8.6) -- совместимость без миграции
- `src/f0-profile.ts` (новый) -- список вопросов Части A (id, текст, пример, 🔑-флаг, тип: text/number/tops/choice/file-ok), `nextProfileQuestion()`, `applyProfileAnswer()`, `isMinimumComplete()`, рендер вопроса с прогрессом -- чистые функции по образцу `f0-fill.ts`, bot.ts остаётся тонким
- `src/bot.ts` -- `/newclient` стартует phase `profile`; обработка ответов профиля (текст, цикл топов кнопками, decision maker кнопками из введённых топов, приоритеты A4.6 кнопками); persist после каждого ответа; `/skip` по правилам матрицы; после минимума — экран «➕ Расширенный / Дальше»; завершение → переход в существующий `collecting`; `/resume`+рестарт восстанавливают текущий вопрос; `/status` в онбординге показывает собранный профиль -- ядро story
- `src/f0-client-card.ts` -- `buildClientCard` переносит `session.profile` в карточку; `renderClientCardMessage` компактно показывает профиль (компания, суть, топы, DM, счётчик заполненности расширенного; ≤15 строк); кнопка «➕ Дозаполнить профиль» на карточке готового клиента запускает расширенные вопросы (тот же механизм персиста, лёгкое состояние) и дописывает `card.json` -- «доступен и позже из карточки»
- `src/f0-profile.test.ts` + дополнения `bot.test.ts`, `f0-client-card.test.ts` -- юнит-тесты сценариев I/O-матрицы: 🔑-gating, /skip-правила, рестарт-резюме, валидность старого persisted JSON без новых полей, рендер профиля в карточке -- регресс-защита

**Acceptance Criteria:**
- Given новый `/newclient`, when трекер отвечает на 🔑-вопросы, then до завершения минимума бот не предлагает способ онбординга и не принимает документы стратегии, а после A3.3 предлагает «➕ Расширенный профиль / Дальше»
- Given phase `profile` и рестарт процесса бота, when трекер шлёт `/resume` или очередной ответ, then бот продолжает с того же вопроса с тем же прогрессом и ранее данные ответы сохранены в сессии
- Given завершённый онбординг с профилем, when трекер открывает `/status`/карточку клиента, then карточка показывает данные профиля и кнопку дозаполнения расширенной части
- Given persisted-сессия формата до 9.1, when бот её восстанавливает, then поведение прежних фаз не изменилось; весь vitest + tsc + canary зелёные

## Design Notes

- **🔑 нельзя пропустить** (в отличие от правила «пропуск 🔑 → 🔴» вопросника): то правило — про Часть B/дозаполнение; для профиля AC1 эпика однозначен — без минимума способ онбординга не предлагается, а 🔴-дозаполнения до extraction ещё не существует. `/skip` на 🔑 → пояснение + повтор.
- **Отдельная очередь, общие паттерны:** профиль живёт до `extraction`, поэтому не встраивается в `F0Gap`; переиспользуются паттерны 7.3/8.6 — persist каждого ответа, header блока, `(i/N)`, мягкая числовая валидация, callback data `action:id`.
- **Подсказки в 9.1** — «текстом · /skip» (без обещания голоса — он появится в 9.5; подсказка 📎 — только в A3.1).

# Story 0.5: Инструкция трекера (Runbook Азизы)

Status: ready-for-dev (2026-04-21)

## Story (Пользовательская история)

Как **коуч практики (Азиза)**,
Я хочу **получить чеклист и инструкцию по работе с ботом**,
Чтобы **я знаю workflow и могу работать автономно с Day 1**.

## Критерии приёмки

1. **Сценарий: Подключение нового клиента**
   ```
   Дано новый клиент подключается
   Когда Азиза открывает runbook
   Тогда описан пошаговый чеклист: получить согласие (Story 0.4), заполнить stakeholder-map, настроить Google Sheets, первая тестовая встреча
     И каждый шаг имеет критерий «сделано»
   ```

2. **Сценарий: Обычный цикл работы с отчётом**
   ```
   Дано встреча завершилась
   Когда Азиза возвращается к работе с ботом
   Тогда описано: `/report` команда, ожидание pipeline (3-5 мин), approve/edit/reject flow
     И описан Bot Menu (📋 Повестка, 📊 Прошлые отчёты, ⚙ Настройки — по мере реализации)
     И описан формат ссылки (Google Drive share / Zoom Cloud URL)
   ```

3. **Сценарий: Проблемные ситуации**
   ```
   Дано pipeline даёт ошибку (таймаут Soniox, API-лимит Claude, неверная ссылка)
   Когда Азиза сталкивается со сбоем
   Тогда runbook содержит fallback:
     - Что написать боту (retry команду)
     - Когда эскалировать Тимуру (Telegram @timur)
     - Как работать без бота (ручной отчёт из транскрипта)
   ```

4. **Сценарий: Правило чувствительности контента**
   ```
   Дано в отчёте есть чувствительный контент (персональные конфликты, зарплатные вопросы, коммерческая тайна)
   Когда Азиза делает review ДО approve
   Тогда описан чеклист чувствительности: что удалять/обобщать перед отправкой клиенту
     И указан red-flag список (ФИО в конфликтах, конкретные суммы, упоминания третьих лиц)
   ```

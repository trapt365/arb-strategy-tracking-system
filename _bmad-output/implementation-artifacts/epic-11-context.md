# Epic 11 Context: Живой прогон WP-39 — устойчивость ядра на свежем клиенте (13 июля)

## Goal

Сделать чистый повторный прогон трёх ядровых отчётов (встреча→отчёт, недельный, трекер гипотез) на любом свежем клиенте без ручного вмешательства. По итогам живого прогона 13 июля зафиксированы три блокера, которые не дают показать «рабочую версию» Айдару: бот падает на файле >20 МБ (нет `bot.catch`), один пустой `owner` в `_okr` роняет весь отчёт (ZodError), клиентские таблицы, созданные под user-OAuth, не читаются под сервис-аккаунтом (403). Помимо блокеров эпик закрывает острые UX-занозы прогона: ложные советы при ошибке API, строгий формат-ретрай участника A3, полотно черновика, отсутствие приёма готового транскрипта. Плюс полировка терминов, security-редакция токена и ребрендинг.

## Stories

- Story 11.1: [P0, блокер] Глобальный `bot.catch` + graceful-обработка аудио >20 МБ
- Story 11.2: [P0, блокер] Устойчивый парсер отчёта — пустой `owner` не роняет весь пайплайн
- Story 11.3: [P1, блокер] Единая read/write identity — клиентские таблицы расшарены на сервис-аккаунт
- Story 11.4: [P1] Классификация ошибок Claude API — честное сообщение вместо «убери файлы»
- Story 11.5: [P1] LLM-экстракция участника профиля (A3) вместо строгого формат-ретрая
- Story 11.6: [P2] Компактная доставка черновика онбординга — резюме вместо полотна
- Story 11.7: [Фича/баг] `/report` принимает готовый транскрипт (md/txt/docx)
- Story 11.8: [Фича] Пакетный приём команды и front-load онбординга — детект типа + предзаполнение + review
- Story 11.9: [P2/полировка] Термины «участники», убрать decision-maker, security-редакция токена, ребрендинг username

## Requirements & Constraints

**Новые FR этого эпика (FR119–FR127):**

- **FR119 [блокер]:** глобальный `bot.catch` обязателен — любая необработанная ошибка логируется, процесс не убивается. Файлы/аудио >20 МБ (лимит Bot API) проверяются по `file_size` до вызова `getFile`; превышение — graceful-деградация с понятным сообщением.
- **FR120 [блокер]:** `parseOkrs` терпит пустой/некорректный `owner` и прочие необязательные поля строки (default «—» + warn, не throw). Онбординг при создании KR без владельца проставляет заглушку-плейсхолдер, не пустую ячейку.
- **FR121 [блокер]:** при создании клиентской таблицы через user-OAuth она автоматически расшаривается на email сервис-аккаунта `geonline-tracking-bot@geonline-report-bot.iam.gserviceaccount.com` (editor). Все read-пайплайны (F1: `_okr`, `_stakeholder_map`, `_f5_metrics`; F5: `_hypotheses`) должны работать на новом клиенте без ручного вмешательства.
- **FR122 [MVP]:** ошибки Claude API классифицируются по `httpStatus`/`anthropicErrorType`: billing/credit → сообщение «закончились кредиты, напиши администратору»; rate-limit → «повтори позже»; too-large-context → совет про файлы. Совет «убери лишние файлы» не показывается, если файл один или контекст заведомо не велик.
- **FR123 [MVP]:** извлечение полей участника `{имя, должность, полномочия, зона}` делается через LLM-экстракцию из естественной фразы (как в приёме стратегии F0). Недостающие поля — точечный доспрос, не hард-ретрай всего ответа.
- **FR124 [MVP]:** доставка черновика онбординга — только компактное резюме (счётчики целей/KR/гипотез/участников + 1–2 строки «что дозаполнить» + гиперссылка на Google Sheet). Детальный перечень неполных KR остаётся в таблице, не в чате.
- **FR125 [MVP]:** `/report` принимает md/txt/docx-транскрипт как вход наравне с аудио. Детект «транскрипт встречи» vs «документ онбординга» гарантирует, что md-транскрипт при активном клиенте не перехватывается флоу онбординга.
- **FR126 [рост]:** детект типа входного документа (фото/OCR, аудио, xlsx/docx/pdf) + маршрутизация в нужный парсер; front-load вопросника в начале онбординга; review-режим (принять/править) вместо строгого Q&A.
- **FR127 [полировка]:** «топы» → «участники работы с трекером» по всему онбордингу; убрать вопрос `a3_3`; редактировать bot-токен из логируемого ctx; BotFather-ребрендинг — Manual TODO Тимуру.

**Регресс-guardrail всего эпика:** после каждой story — canary + golden + весь vitest + tsc зелёные; `clientId === 'geonline'` fallback цел; `GEONLINE_F0_SHEET_ID` не затронут; боевая таблица Geonline не изменена.

**Внешний блокер (не код):** баланс Anthropic API = 0 на прогоне 13 июля — ядро нельзя проверить вживую до пополнения. COGS LLM ~50k тг/мес учитывать в ценообразовании.

## Technical Decisions

**Ключевые точки изменений в коде:**

- `bot.js` — добавить `bot.catch` (глобальный обработчик); добавить проверку `file_size` в `handleMeetingFileIntake` (bot.js:3535) до вызова `getFile`; переделать доставку черновика после `runF0FullDraft` (bot.js:2380).
- `adapters/sheets.js` — `parseOkrs` (sheets.js:486): заменить жёсткую zod-валидацию `owner` на `optional` + default «—»; `readHypothesesSheet` (sheets.js:450): 403 сейчас блокирует F5 трекер.
- `f0-onboarding.js` — `runF0FullDraft` (f0-onboarding.js:80): пробросить классифицированный тип ошибки; LLM-экстракция участника вместо формат-парсера.
- `claude.js` — классификация ошибок по `httpStatus`/`anthropicErrorType` (claude.js:209).
- Создание таблицы при онбординге: после `spreadsheets.create` под user-OAuth вызвать `permissions.create` для email сервис-аккаунта (role: editor) — это устраняет 403 на всех read-пайплайнах для новых клиентов.

**Архитектурные инварианты (из Epic 7, без изменений):**
- Мультиклиентность через `clientId`; Geonline изолирован как canary-клиент.
- Read/write через единый sheets adapter; camelCase внутри кода, snake_case на уровне Sheets.
- Claude API вызовы только из `claude.js`; retry 3 попытки, exponential backoff.
- F1 pipeline: `_okr` + `_stakeholder_map` + `_f5_metrics` → chain of prompts → отчёт. F5 pipeline: `_hypotheses` → трекер.

**Детект транскрипт vs онбординг-документ (FR125):** md/txt/docx при активном клиенте анализируется эвристически (наличие таймкодов, реплик, признаков протокола встречи) перед решением о маршрутизации в F1 или F0.

## UX & Interaction Patterns

- При файле >20 МБ бот явно сообщает о лимите и предлагает альтернативу (сжать/разбить), вместо тишины или краша.
- Ошибка Claude API отображается по типу: пользователь никогда не получает вводящий в заблуждение совет («убери файлы»), когда проблема в кредитах или rate-limit.
- Участник онбординга (A3) вводится свободной фразой; бот подтверждает извлечённые поля и доспрашивает только недостающее — не отвергает весь ввод.
- Черновик онбординга: одно компактное сообщение с итоговыми счётчиками и ссылкой; детальный список пробелов — в Google Sheet, не в чат.
- Термин «топы» устранён полностью; вопрос `a3_3` удалён из вопросника.

## Cross-Story Dependencies

- **11.1 → 11.2 → 11.3** — строгий порядок: блокеры ядра фиксируются первыми. После 11.3 возможен чистый повторный прогон трёх отчётов на свежем клиенте.
- **11.3** (расшивка прав) влияет на все последующие read-пайплайны (F1 отчёт, F5 трекер) для новых клиентов — без неё 11.4–11.9 нельзя проверить end-to-end на не-Geonline клиенте.
- **11.5** (LLM-экстракция участника) и **11.8** (пакетный приём + front-load) перекрываются по зоне изменений в F0-онбординге — координировать, чтобы не конфликтовать в `f0-onboarding.js`.
- **11.7** (приём транскрипта) зависит от корректного детекта типа документа, который также затрагивается в **11.8** (детект input-type); унифицировать логику детекта в одном месте.
- Регресс Geonline после каждой story — обязательная проверка перед переходом к следующей.

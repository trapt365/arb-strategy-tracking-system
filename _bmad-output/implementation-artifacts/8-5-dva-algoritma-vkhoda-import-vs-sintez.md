# Story 8.5: Два алгоритма входа онбординга — импорт готовой стратегии vs синтез

Status: done  # 2026-07-09: a15d121 (dev) + 498bfce (фиксы ревью); 559/559 vitest, tsc, build, canary --no-claude PASS

## Решение (утверждённая дизайн-записка)

Полный текст: `_bmad-output/planning-artifacts/design-f0-import-vs-synthesis-2026-07-08.md`
(статус УТВЕРЖДЕНО ТИМУРОМ 2026-07-09). Ключевое:

- **Импорт — второй продюсер `F0FullExtraction`, не параллельный пайплайн.** Один `.xlsx`
  → `src/f0-import.ts` (0 LLM-вызовов) → тот же черновик → общий даунстрим (саммари,
  инварианты 1–3, диалог дозаполнения 7.3/8.6, `/confirm` → Sheets) без правок.
- **Развилка:** кнопки «📥 Есть готовая стратегия (Excel)» / «🧠 Собрать из документов»
  при `/newclient` + автодетект по расширению первого файла. Пути в одной сессии
  не смешиваются; отказ импорта → честный текст + предложение синтеза.
- **Маппинг:** формат A («наша» таблица, детект по листу `_okr`/знакомым заголовкам,
  обратный к `mapOkrRows`/`mapStakeholderRows`/`mapHypothesisRows`) + формат B
  (произвольная таблица: лист с максимумом совпавших заголовков-синонимов, порог ≥ 3,
  формулировка KR обязательна). Незамапленное → `extraction.unrecognized`. Пустое не
  выдумываем (инвариант 3). `kr_type` всегда `'metric'`.

## Ответы Тимура на открытые вопросы записки (2026-07-09)

1. Фикстуры — **только Geonline** (реальных файлов других клиентов не используем).
2. Стратегия текстом — через существующий синтез-путь: **ок**.
3. Автодетект по расширению + необязательные кнопки: **ок**.
4. Гипотезы при импорте не синтезируем, но **добавить кнопку** «🧠 Досинтезировать
   гипотезы из этого же файла» после импорта (LLM по текстифицированному xlsx,
   существующий промпт `f0-full-extraction`, берём только hypotheses).

## Заметки исполнения

- **Рефакторинг buildF0Draft:** общий хвост (persist черновика, саммари, переход в
  filling, первый вопрос) вынесен в `deliverF0Draft` — импорт и синтез идут одним
  даунстримом; ревьюер построчно сверил со старым кодом, деградаций нет.
- **Досинтез гипотез:** переиспользован промпт `f0-full-extraction` (берём только
  hypotheses, форсим `synthesized:true` — «через LLM ⇒ требует подтверждения»).
  Кнопка предлагается когда в файле 0 гипотез; вопросы метрик — в конец очереди,
  прогресс диалога не сдвигается. `importSourceText` персистится (кнопка живёт
  после рестарта); при >400k символов не сохраняется — кнопка честно недоступна.
- **Ограничения формата B (приняты):** `kr_type` всегда metric; TZ-дрейф дат из
  Date-ячеек ±1 день; company не извлекается (null → спросит /confirm-поток как
  обычно). Потолок листа 2000×100 — реальные стратегии на порядки меньше.

## Мини-ревью (2026-07-09, свежий контекст по a15d121)

0 HIGH / 3 MED / 3 LOW — все закрыты в 498bfce (LOW-2 частично, см. выше):
1. **MED-1:** явная кнопка «📥 Импорт» фиксировала mode до файла, отказ импорта не
   сбрасывал → тупик «пришли документы» / «документы не добавляются». Фикс: сброс
   mode в catch, если ничего не принято.
2. **MED-2:** без forward-fill направления merged-ячейки дробили objectives на
   фиктивные цели с именем листа (молча уезжало в okr_group Sheets). Фикс: тянем
   последнее непустое направление вниз по группе.
3. **MED-3 (воспроизведено ревьюером):** 16-КБ xlsx с раздутым used range (~1M строк)
   вешал event loop на ~47 с / 1.2 ГБ heap — бот однопоточный, вставал и F1-цикл.
   Фикс: потолок 2000×100 + частичный проход для детекта/скана; теперь ~0.2 с.
4. **LOW-1/2/3:** guard-текст «принят Excel», процентные ячейки «9%» вместо 0.09,
   тексты ошибок для xlsx.

Проверено и отклонено ревьюером: рестарт между приёмом xlsx и /draft (collecting не
персистится — консистентно с потерей пакета документов); совместимость persist
(AIPLUS filling восстанавливается, откат на пре-8.5 код безопасен); гонки f0_synth_hypo
(последовательный polling + processing-флаг); двойное нажатие; 4096-лимиты; warn-only;
round-trip формата A с mapOkrRows/mapStakeholderRows/mapHypothesisRows.

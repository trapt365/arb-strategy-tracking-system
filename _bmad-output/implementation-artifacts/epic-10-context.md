# Epic 10 Context: Разворот к ядру — три отчёта, прощающий онбординг (WP-39 Ф2)

<!-- Generated from planning artifacts. Regenerate with compile-epic-context if planning docs change. -->

## Goal

Epic 9 закрыл все stories по коду, но живой прогон пилота вскрыл стратегический дрифт: продукт построил систему управления клиентами, а ядро трекера — три отчёта — не работает целиком. Epic 10 возвращает фокус: (1) отчёт после встречи, (2) недельный отчёт (главный результат), (3) трекер гипотез уровня geonline. Параллельно устраняются блокирующие баги (grounding, xlsx) и избыточный онбординг. Якорь: рабочая версия для передачи Айдару — трекер заводит клиента без допроса, аудио встречи питает недельный отчёт, все три отчёта проходят golden-eval на эталонных данных geonline нед.8/12.

## Stories

- Story 10.1: Приём аудио встречи существующего клиента → недельный отчёт
- Story 10.2: Прощающий вход клиента; тяжёлый профиль → `/advanced`
- Story 10.3: Багфикс grounding — флаг смешения клиентов
- Story 10.4: Багфикс xlsx — реальный файл ARB v1.1
- Story 10.5: Трекер гипотез — третий тип отчёта
- Story 10.6: Golden-eval отчётов
- Story 10.7: Косметика — маркер клиента, приветствие, длины, мелочи
- Story 10.8: Трекер гипотез — структура уровня geonline

## Requirements & Constraints

**Порядок выполнения:** 10.1 → 10.2 → 10.3 → 10.4 → 10.5 → 10.8 → 10.6 → 10.7. Story 10.8 идёт ДО 10.6 (golden-eval): 10.5 построил F5 плоским, 10.8 заменяет формат на структурный — именно этот формат проверяет 10.6.

**FR112 — Intake аудио встречи (10.1):** Бот принимает запись встречи (`.m4a`, `.ogg`, `.mp4`, ссылку) для активного клиента. Распознаёт входящий файл как встречу, транскрибирует существующим `src/adapters/soniox.ts`, прогоняет существующий F1-конвейер, персистит результат. Недельный отчёт (9.7) учитывает обработанные встречи; неделя без встреч — честное «встреч не обработано», логику 9.7 не изменять.

**FR113 — Прощающий онбординг (10.2):** Минимум `/newclient` сужен до названия компании + сути бизнеса. Топы, decision-maker, расширенный профиль (9.1) — только через `/advanced`. Код 9.1 не удаляется. Старые сессии с полным профилем совместимы (поля optional в Zod).

**FR114 — Grounding-флаг смешения (10.3):** Если имя компании из профиля расходится с сущностями в загруженных документах — бот показывает 🔴/вопрос трекеру, не молчаливо собирает OKR по чужим данным. При совпадении — поведение как раньше.

**FR115 — xlsx ARB v1.1 (10.4):** Файл «ARB Solutions Стратегический трекер v1.1 (1).xlsx» должен маппиться (словарь синонимов заголовков пополнен, файл добавлен в тесты). При отказе — конкретный запасной путь (документы/вопросник), не тупик.

**FR116 + Story 10.8 — Трекер гипотез (10.5 + 10.8):** Третий тип отчёта. Story 10.5 строит базовый pipeline; Story 10.8 заменяет формат на структурный уровня geonline нед.12. Итоговая структура отчёта (жёсткий критерий, НЕ плоский список):
- Шапка: период / «Подготовил: {имя трекера}» / «Для: {клиент}» / «База: снимок нед.{N-1} ({M} гипотез)»
- Легенда 7 статусов (с заглавной: «🟢 Работает», «🟡 В тесте» и т.д.)
- Секции по департаментам: ответственный + метрики; таблица «Обновления нед.N-1→N» (колонки: `#|Гипотеза|Статус N-1|Статус N|Комментарий`); таблица «🆕 Новые» (колонки: `#|Гипотеза|Статус|Запуск|Результат|Следующий шаг`)
- Сводная матрица по департаментам
- Ключевые выводы
- Методологическая сноска

Строка «Период» — диапазон недель «Недели 1–{N}» если известен, иначе даты встреч. Богатые ячейки (комментарий/запуск/результат/следующий шаг) синтезируются LLM из F1-отчётов недели (тот же источник, что 9.7); при отсутствии встреч — структура строится из листа+дельты без комментариев. Снимок гипотез и номер недели хранятся в персисте клиента; первая неделя без базы — полный список без дельты. Доставка: полный трекер — вложением `.md` (`sendDocument`); в чат — компактная сводка (матрица по департаментам + топ-3 сигнала + «полный во вложении»). Не резать через `splitForTelegram`.

**FR117 — Golden-eval (10.6):** Недельный отчёт (9.7) и трекер гипотез (FR116/10.8) прогоняются на входах geonline нед.8/12 (транскрипты + снимок Excel-трекинга) и сравниваются с эталонами (`Geonline отчет о трекшн неделе 8.md`, `трекер гипотез неделя 12.md`) по структуре/полноте. Eval-скрипт воспроизводим (`scripts/`), входы — в фикстуры; боевую таблицу Geonline не трогать. При провале — отчёт «чего не хватает» для доработки промпта/логики.

**FR118 — Косметика (10.7):** Маркер активного клиента виден постоянно. Приветствие/название бота обновлены. Имена персональных листов Sheets сокращены до имени топа. Устранены: `/start`-дубли, залипший онбординг-стейт, повторяющийся футер, противоречивые маркеры готовности, «Скоро»-пункты в help. Ни одна пользовательская выдача не длиннее ~15 строк без ссылки.

**Регресс-guardrail всего эпика:** Инварианты Epic 7 (1–4) без изменений. `clientId === 'geonline'` fallback и `GEONLINE_F0_SHEET_ID` не ломать. Canary + golden + весь vitest + tsc зелёные после каждой story. Боевая таблица Geonline не затрагивается. Stories 10.5/10.6 — под присмотром: критерий приёмки — golden-eval, не только «зелёные тесты».

## Technical Decisions

**Существующая инфраструктура используется без изменений:** `src/adapters/soniox.ts` — Soniox-транскрипция; F1-конвейер — 4-шаговая цепочка extraction → analysis → formatting → delivery; `types.ts` Zod-схемы. Story 10.1 добавляет только intake-обработчик аудио-файлов поверх этого.

**Один файл на pipeline.** Трекер гипотез (10.5/10.8) — новый pipeline-файл (аналогично `f3-lite.ts`). Шаблон — в `prompts/` как `.md` файл. Старый плоский форматтер 10.5 сохраняется как `formatHypoReportFlat` (не удалять).

**Персист недельного снимка гипотез** — JSON-файл под `data/{client_id}/` (паттерн ADR-003). Схема: `{weekNumber, hypotheses: [{dept, okr, status, ...}]}`. `computeDelta`/снимок переиспользуются; старый снимок не роняет пайплайн.

**Карта статусов (`STATUS_EMOJI`) пополнена:** «завершена→🟢», «приостановлена/⏸️→⛔», «идёт/in progress→🟡»; неизвестный статус — `⬜`. Статусы в ячейках — с заглавной буквы.

**Grounding (9.2)** переиспользуется в 10.5/10.8 для имён/названий из профиля. В 10.3 расширяется: сравнение имени профиля с сущностями загруженных документов → флаг смешения.

**Доставка трекера гипотез:** `sendDocument` (`.md`-вложение) + компактная сводка в чат. Не `splitForTelegram`.

**Enforcement rules без исключений:** `loadPrompt()`, `parseClaudeJSON(raw, Schema)`, `readSheet()` по имени заголовка, `withRetry()` для Claude-вызовов, `pino` с `{pipeline, step, clientId}`.

**Клиентская изоляция** обязательна во всех stories. Intake, трекер гипотез, golden-eval — только по активному клиенту.

## UX & Interaction Patterns

**Прощающий вход (10.2):** онбординг — «пришли стратегию — что есть», без допроса. `/advanced` доступен и при онбординге, и позже из карточки клиента. Недостающее показывается в таблице, не вытягивается допросом в чате.

**Intake встречи (10.1):** бот сам определяет входящий файл как встречу и привязывает к активному клиенту. Immediate acknowledgment (< 2 сек) — подтверждение что принято, до завершения транскрипции.

**Флаг смешения (10.3):** вопрос трекеру формулируется конкретно — «документы про X, клиент Y — чьи данные берём?» с вариантами-кнопками.

**Маркер клиента (10.7):** активный клиент виден постоянно в заголовке каждого отчёта/повестки. Ни одна выдача не длиннее ~15 строк без ссылки.

**Доставка трекера гипотез (10.8):** в чат — компактная сводка (матрица + топ-3 сигнала + «полный во вложении»), полный документ — `.md`-файлом.

## Cross-Story Dependencies

**10.1 блокирует 10.5, 10.8 и 10.6:** трекер гипотез синтезирует комментарии из F1-отчётов встреч; golden-eval требует рабочего аудио-intake на входах geonline нед.8/12.

**10.3 используется в 10.5/10.8:** grounding должен корректно работать до внедрения трекера гипотез (имена из профиля, флаг смешения).

**10.5 → 10.8 → 10.6:** 10.8 заменяет формат 10.5 на структурный; golden-eval (10.6) проверяет именно структурный формат 10.8. Порядок строгий.

**10.6 — критерий приёмки 10.5/10.8:** golden-eval — не отдельная опция, а обязательный acceptance criterion для трекера гипотез.

**Обратная совместимость 10.2 с 9.1:** profile-поля в `ClientCard` становятся optional; существующие клиенты с полным профилем работают без изменений; grounding (9.2) работает с обоими вариантами профиля.

**Инварианты Epic 7 (1–4)** сквозные для всего эпика: любая story, затрагивающая client-routing или Geonline-пути, верифицирует инварианты перед сдачей.

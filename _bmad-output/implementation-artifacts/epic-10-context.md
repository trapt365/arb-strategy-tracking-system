# Epic 10 Context: Разворот к ядру — три отчёта, прощающий онбординг (WP-39 Ф2)

<!-- Generated from planning artifacts. Regenerate with compile-epic-context if planning docs change. -->

## Goal

Epic 9 закрыл все 7 stories по коду, но живой прогон пилота вскрыл стратегический дрифт: продукт построил систему управления клиентами, а ядро трекера — три отчёта — не работает целиком. Epic 10 возвращает фокус: (1) отчёт после встречи, (2) недельный отчёт (главный результат), (3) трекер гипотез. Параллельно устраняются блокирующие баги (grounding, xlsx) и избыточный онбординг. Якорь: рабочая версия для передачи Айдару — трекер заводит клиента без допроса, аудио встречи питает недельный отчёт, все три отчёта проходят golden-eval на etalon-данных geonline.

## Stories

- Story 10.1: Приём аудио встречи существующего клиента → недельный отчёт
- Story 10.2: Прощающий вход клиента; тяжёлый профиль → `/advanced`
- Story 10.3: Багфикс grounding — флаг смешения клиентов
- Story 10.4: Багфикс xlsx — реальный файл ARB v1.1
- Story 10.5: Трекер гипотез — третий тип отчёта
- Story 10.6: Golden-eval отчётов
- Story 10.7: Косметика — маркер клиента, приветствие, длины, мелочи

## Requirements & Constraints

**FR112 — Intake аудио встречи (Story 10.1):** Бот принимает файл записи встречи (`.m4a`, `.ogg`, `.mp4`, ссылку) для активного/выбранного клиента, распознаёт его как встречу, транскрибирует существующим `src/adapters/soniox.ts`, прогоняет существующий F1-конвейер и персистит результат. Недельный отчёт (9.7) должен учитывать обработанные встречи. Неделя без встреч — честное «встреч не обработано», логику 9.7 не трогать.

**FR113 — Прощающий онбординг (Story 10.2):** Минимум `/newclient` сужен до названия компании + сути бизнеса. Топы, decision-maker, расширенный профиль (Story 9.1) — только через `/advanced`. Код 9.1 не удаляется, вынесен из основного потока. Старые сессии с полным профилем совместимы (поля optional в Zod).

**FR114 — Grounding-флаг смешения (Story 10.3):** Если имя компании из профиля расходится с сущностями в загруженных документах — бот показывает 🔴/вопрос трекеру («документы про X, клиент Y»), не молчаливо собирает OKR по чужим данным. При совпадении — поведение без изменений.

**FR115 — xlsx ARB v1.1 (Story 10.4):** Файл «ARB Solutions Стратегический трекер v1.1 (1).xlsx» должен маппиться (словарь синонимов заголовков пополнен). При отказе — конкретный запасной путь (документы/вопросник), не тупик.

**FR116 — Трекер гипотез (Story 10.5):** Третий тип отчёта, шаблон geonline нед.12: гипотезы по департаментам/OKR → таблица обновлений статусов (прошлая→эта неделя) + новые гипотезы + сводная таблица по статусам + ключевые выводы. Строится как «прошлая неделя + дельта»: бот хранит недельный снимок гипотез и номер недели в персисте клиента. Первая неделя без базы — полный список без дельты. Имена/названия — из профиля (grounding 9.2).

**FR117 — Golden-eval (Story 10.6):** Недельный отчёт (9.7) и трекер гипотез (FR116) прогоняются на входах geonline нед.8/12 (транскрипты + снимок Excel) и сравниваются с эталонами практики по структуре/полноте (все секции шаблона присутствуют, ключевые сущности покрыты). Eval-скрипт воспроизводим (`scripts/`), входы — в фикстуры; боевую таблицу Geonline не трогать.

**FR118 — Косметика (Story 10.7):** Маркер активного клиента виден постоянно; приветствие/название бота обновлены; имена персональных листов Sheets сокращены до имени топа; устранены `/start`-дубли, залипший онбординг-стейт, повторяющийся футер, противоречивые маркеры готовности, «Скоро»-пункты в help.

**Порядок выполнения:** 10.1 → 10.2 → 10.3 → 10.4 → 10.5 → 10.6 → 10.7. Спина (10.1) — первой: без приёма аудио недельный отчёт пуст.

## Technical Decisions

**Существующая инфраструктура используется без изменений:** `src/adapters/soniox.ts` — Soniox-транскрипция; F1-конвейер — 4-шаговая цепочка extraction → analysis → formatting → delivery; `types.ts` Zod-схемы. Story 10.1 добавляет только intake-обработчик аудио-файлов поверх этого.

**Модульный монолит: один файл на pipeline.** Трекер гипотез (10.5) — новый pipeline-файл (аналогично `f3-lite.ts`). Шаблон берётся из `prompts/` в `.md` файле.

**Клиентская изоляция обязательна во всех stories.** Intake встречи, трекер гипотез, golden-eval — только по активному/выбранному клиенту. `clientId === 'geonline'` fallback и `GEONLINE_F0_SHEET_ID` не ломать.

**Персист недельного снимка гипотез** нужен для дельта-логики 10.5. Хранится в JSON-файлах под `data/{client_id}/` (паттерн ADR-003). Схема: `{weekNumber, hypotheses: [{dept, okr, status, ...}]}`.

**Grounding (9.2)** переиспользуется в 10.5 для imён/названий из профиля клиента. В 10.3 расширяется: сравнение имени профиля с сущностями загруженных документов → флаг смешения.

**Доставка трекера гипотез** — компактная (паттерн 8.3): саммари + ссылка на полный артефакт. Не в одном сообщении целиком.

**Enforcement rules действуют без исключений:** `loadPrompt()` для промптов, `parseClaudeJSON(raw, Schema)` для Claude output, `readSheet()` по имени заголовка (не по индексу), `withRetry()` для Claude-вызовов, `pino` logger с `{pipeline, step, clientId}`.

**Регресс-guardrail после каждой story:** canary + golden + весь vitest + tsc зелёные. Боевая таблица Geonline не затрагивается. Stories 10.5 и 10.6 — под присмотром: критерий приёмки качества — golden-eval, не только «зелёные тесты».

## UX & Interaction Patterns

**Прощающий вход (10.2):** онбординг — это «пришли стратегию — что есть», без допроса. Сразу к делу. `/advanced` доступен и в момент онбординга, и позже из карточки клиента. Тон: союзник, не судья — не наказывать за отсутствие данных, показывать недостающее в таблице.

**Intake встречи (10.1):** бот сам определяет, что входящий файл = встреча, и к какому клиенту относится (активный клиент). Immediate acknowledgment (< 2 сек) — подтверждение что принято, без ожидания транскрипции.

**Флаг смешения (10.3):** сообщение трекеру показывает конкретику — «документы про X, клиент Y — чьи данные берём?» с вариантами ответа (кнопки). Не технический алерт, а вопрос с контекстом.

**Маркер клиента (10.7):** активный клиент должен быть виден постоянно — в заголовке каждого отчёта/повестки/команды. Ни одна пользовательская выдача не должна быть длиннее ~15 строк без ссылки.

## Cross-Story Dependencies

**10.1 блокирует 10.5 и 10.6:** трекер гипотез строится по обработанным встречам; golden-eval требует, чтобы аудио-intake работал на входах geonline нед.8/12.

**10.3 используется в 10.5:** grounding профиля (имена/сущности) должен работать корректно до внедрения трекера гипотез.

**10.6 является критерием приёмки 10.5:** golden-eval — не отдельная опция, а обязательный acceptance criterion для трекера гипотез. Запускать 10.6 параллельно с финализацией 10.5.

**Инварианты Epic 7 (1–4) сквозные для всего эпика:** любая story, которая трогает client-routing или Geonline-пути, должна верифицировать эти инварианты перед сдачей.

**Обратная совместимость 10.2 с 9.1:** profile-поля в `ClientCard` становятся optional, но существующие клиенты с полным профилем продолжают работать без изменений. Grounding (9.2) работает и с минимальным профилем.

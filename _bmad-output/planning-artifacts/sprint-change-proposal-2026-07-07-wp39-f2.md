# Sprint Change Proposal — 2026-07-07 (WP-39 Ф2: Feature F0 «Онбординг клиента»)

**Триггер:** Стартовое задание WP-39 Ф2 (DS-strategy/inbox/wp-39/f2-kickoff.md, источники promise-scenarios.md v1.1 + onboarding-data-map.md v1.1).
**Scope:** Новый Epic 7 — Feature F0 «Онбординг клиента через бота» + минимальная мультиклиентность. Direct Adjustment: существующие эпики не переписываются.

---

## 1. Суть изменения

Закрываем дыру «Week 0: данные загружаются в систему вручную Тимуром — каждый новый клиент требует архитектора».

**Обещание F0 (контракт):** «Дай боту артефакты готовой стратегии — через одну сессию вопросов у трекера есть рабочая среда: заполненный Google Sheets, банк гипотез, участники, календарь. Можно проводить первую трекшн-встречу».

| Параметр | Значение |
|----------|---------|
| Триггер | Трекер: команда «новый клиент» в Telegram |
| Входы | Файлы артефактов стратегии (md/docx/pdf): итоговый протокол сессии, OKR-документ, нарратив; опционально контекст-файл |
| Процесс | Извлечение → черновик трекеру → вопросы ТОЛЬКО про отсутствующее (базы KR, ответственные, контакты, слоты) → подтверждение |
| Выходы | (1) Google Sheets «Стратегический трекинг» по шаблону Geonline v2.0; (2) карточка клиента в боте; (3) чеклист готовности к неделе 1 |
| Время | Черновик из артефактов ≤ 15 мин; полный онбординг ≤ 1 рабочий день |

**Инварианты (блокирующие):**
1. KR без числовой базы «с X до Y» и ответственного не выпускается из онбординга — бот помечает 🔴 и требует дозаполнения (урок Geonline: 12 из 30 KR несчитаемы).
2. Каждая гипотеза имеет метрику проверки.
3. Не выдумывать: не извлёк → показать нераспознанное, спросить трекера. Нет артефактов → ручной ввод по короткой форме.
4. Ничего не уходит клиенту без одобрения трекера.

## 2. Уточнение терминологии F0

В PRD «F0» исторически означал ручное заполнение стейкхолдерной карты (~30 мин, Story 0.2 — done для Geonline). WP-39 переопределяет F0 как **фичу бота**: автоматизацию того же онбординга из артефактов стратегии. Epic 7 автоматизирует и заменяет ручной процесс Story 0.2; FR20/FR55 (ручное заполнение) остаются как fallback «нет артефактов → короткая форма».

## 3. Зафиксированное расхождение с kickoff

Kickoff утверждает «бот уже реализует недельный цикл F1/F4/F3/F5, запущен end-to-end 25.05.2026». По репозиторию реализован **только F1** (Epic 1, stories 1.1–1.11): в src/ нет F4/F3/F5-конвейеров, Epic 3/4 — backlog, Epic 2 — deferred-growth (решение 15.04.2026); `prompts/agenda.md` (F4) протестирован в Story 0.3, но в прод-код не встроен.

**Следствие для критерия приёмки Ф2 №3** («существующий недельный цикл работает для обоих клиентов»): в scope Ф2 трактуем как «**F1 работает для обоих клиентов** (бот по транскрипту понимает, чей клиент и чья встреча)»; F4/F3 получают мультиклиентность при их реализации (Epic 3/4) — фундамент (реестр клиентов, резолвинг клиента из чата) закладывает Epic 7. Подтвердить у Тимура.

## 5. Порядок реализации (вертикальный срез первым)

```
7.1 OKR-документ → черновик панели OKR (вертикальный срез)
7.2 Полный вход: docx/pdf, несколько файлов, гипотезы + участники (тест на SAM)
7.3 Диалог дозаполнения: вопросы только про отсутствующее + подтверждение
7.4 Создание Google Sheets по шаблону «Стратегический трекинг v2.0»
7.5 Карточка клиента + чеклист готовности к неделе 1
7.6 Минимальная мультиклиентность + регресс Geonline
```

Формат stories — лёгкий (solo MVP): 3 AC, ~5 задач, 1 проход ревью. Полный BMad-цикл не разворачивается.

## 6. Критерии приёмки Ф2 (из kickoff)

1. Азиза самостоятельно онбордит 2 пилотных клиентов через бота.
2. Черновик из артефактов ≤ 15 мин; инварианты 1–2 соблюдены (проверка: OKR-документ с KR без базы → бот блокирует).
3. F1 работает для обоих клиентов (см. п.3 о трактовке).
4. Регресс не сломан: Geonline работает как раньше.

## 8. Выводы из эталонных файлов (для промптов извлечения)

- Форматы входов радикально разные: GeOnline — консалтинговая проза с таблицами, гипотезы местами уже в IF-THEN-BECAUSE; SAM-протокол — жёсткие секции с ID (D/A/OQ/C), гипотез нет вообще, baseline нет (прямо вынесен в открытый вопрос). Промпт — контент-ориентированный (искать сущности), не layout-ориентированный; гипотезы при отсутствии — синтезировать из решений/инициатив с пометкой «требует подтверждения трекером».
- Контактов (telegram/телефоны) и расписания трекшн-встреч нет НИ в одном эталоне → это всегда вопросы диалога дозаполнения.
- Технические требования пайплайна: срезать мусорные vscode-remote URL из оглавлений (тысячи токенов), терпеть RU/KZ code-switching в ячейках, нормализовать имена (Шыңғыс/Шынгыс) и форматы дат («До 30.07.2026», «Q1 2026», «Постоянно», «Нед. 5»), эмодзи-статусы (🟢🟡🔴⏳⛔🆕) — семантика, не декорация.
- Методология каскада (6 уровней) и форматы гипотезы/задачи — из «Контекст для создания google sheets файла geonline.md» → в системный промпт.

## Executive Summary

Большинство стратегий проваливаются на этапе внедрения — не потому что плохие, а потому что у команды не поставлена **практика стратегического исполнения** — способ работы, при котором команда формулирует гипотезы, проверяет их связь с OKR, тестирует в недельном ритме и принимает решения без зависимости от CEO. ARB Solutions ставит эту практику через еженедельный коучинговый трекинг с живыми трекерами — но модель упирается в потолок: контроль качества завязан на руководителе лично, трекеры перегружены рутиной, клиент не видит прогресс между встречами. Результат — бизнес не масштабируется за 2-3 клиента.

При этом методология трекинга доказала эффективность без AI: клиент Geonline за 5 недель запустил аудит отдела продаж, вывел 3 гипотезы роста в тестирование и впервые провёл общекомандную синхронизацию по OKR. AI-система призвана сделать этот результат воспроизводимым и масштабируемым.

AI-система снимает это ограничение. Трекер, освобождённый от рутины, направляет время на то, что создаёт результат: задавать правильные вопросы, вскрывать системные ограничения, возвращать команду к стратегическому фокусу — и быстрее ставит команде **практику стратегического исполнения**. AI расширяет восприятие трекера по нарастающей: на старте экономит 3-4 часа рутины в неделю; через 2-3 месяца даёт доказательную обратную связь на основе трендов; через 6 месяцев формирует объективную базу для измерения **уровня владения практикой**. AI измеряет наблюдаемые компоненты практики (постановка задач, формулировка гипотез, связь с целями, перепланирование) — но не все: выполнение обязательств и самостоятельность решений происходят между встречами.

Единый data pipeline (транскрипт → AI-анализ → метрики), три output-формата: **трекер** получает автоотчёты, фидбэк по методологии (например: «4 из 5 встреч ты давала советы больше 50% времени — попробуй на следующей неделе больше задавать вопросы») и повестки встреч; **руководитель ARB** — QC-скоринг всех трекеров на одном экране; **CEO-клиент** — видимость прогресса постановки практики: какие гипотезы тестируются, какие задачи двигают стратегию, где слепые зоны. На MVP прозрачность — промежуточная ценность: CEO видит прогресс и продолжает платить. На горизонте 3-6 месяцев прозрачность показывает рост уровня владения практикой — скорость обучения команды. Транскрипты содержат стратегические данные клиента — pipeline спроектирован с разделением доступа по ролям.

Три уровня ценности: AI доставляет **прозрачность** (данные, отчёты, метрики). Трекер использует прозрачность для **постановки практики** (формирование навыков команды). Прозрачность делает **уровень владения практикой** видимым и измеримым — замыкая цикл.

AI обеспечивает **воспроизводимость качества** (каждый трекер, независимо от опыта, доводит клиента до результата) и **институциональную память клиента** (история сессий, тренды поведения, доменный контекст) — преемственность при смене трекера через накопленные данные. Это открывает масштабирование с 2-3 клиентов до 10-20.

MVP: пилот с одним клиентом (Geonline), один трекер (Азиза). Стек: Soniox + Google Sheets + AI pipeline (orchestrated prompts) + Telegram. Срок: апрель-май 2026.

Почему сейчас: LLM впервые способны анализировать неструктурированные разговоры на русском с качеством, достаточным для бизнес-выводов. Год назад — невозможно. Через год — commodity. Окно для создания доменного преимущества: 12-18 месяцев.

### Что делает продукт особенным

**Полный цикл, которого нет ни у кого:** стратсессия → OKR → еженедельный коучинговый трекинг → AI-контроль качества → постановка практики стратегического исполнения у команды клиента. Gong анализирует разговоры без привязки к OKR. Viva Goals трекает цели без контроля качества коучинга. ARB AI-Tracking понимает контекст сессии целиком — кто говорит, из какой роли, про какой OKR, в какой «шляпе» трекер.

**People-first:** живой трекер остаётся в центре — AI делает его быстрее, объективнее и видящим больше.

**Результат — не отчёты, а поставленная практика.** Метрика успеха — команда научилась ставить правильные гипотезы, связывать задачи со стратегическими целями и планировать в недельном ритме. Не эффективность pipeline.

**Конфиденциальность как предусловие.** Транскрипты содержат стратегические данные клиента — кассовые разрывы, кадровые решения, конфликты. Каждая роль видит только свой слой данных. Трекер — единственный, кто видит всё.

## Success Criteria

### Два горизонта метрик

**MVP метрика успеха** (недели 1-6): принятие pipeline — ≥ 19 из 24 отчётов отправлены клиенту. **Стратегическая метрика** (post-Phase 2): скорость обучения команды — тренд качества гипотез.

### User Success

**Трекер (Азиза):**
- Отчёты, отправленные клиенту через pipeline: ≥ 19 из 24 за 3 недели (Азиза доверяет качеству)
- Время на рутину: < 2ч/нед (было 4ч)
- Повестка на понедельник: оценка полезности ≥ 4/5 (qualitative signal)
- Aha-момент: «Отчёт готов — я только проверила. А система показала, что я больше половины встреч давала советы вместо вопросов»

**Руководитель ARB (Айдар):**
- QC-сводка за неделю: Айдар считает полезной (qualitative signal, Phase 2)
- AI показывает распределение шляп и talk ratio **как информацию**, без жёсткого порога — трекер и руководитель решают, что нормально для конкретного контекста
- Покрытие OKR: какие KR обсуждены (≥ 2 реплики по теме), какие пропущены

**CEO-клиент (Дамир):**
- Еженедельное сводное уведомление (5 строк + 🟢🟡🔴 по каждому OKR + один конкретный инсайт/вопрос к CEO). Доставка: понедельник утром
- Вовлечённость: Дамир открыл уведомление ≥ 2 из 3 недель
- Отреагировал (написал трекеру, принял решение, задал вопрос): ≥ 1 из 3 недель

**Уровень владения практикой (стратегическая метрика, lagging, post-Phase 2):**
- **Главная метрика — скорость обучения команды:** тренд улучшения качества гипотез от месяца к месяцу. Важен не абсолют (сколько правильных сегодня), а динамика (как быстро команда учится формулировать лучше — связь с OKR, измеримость, срок проверки)
- Скорость тестирования гипотез: тренд роста с ~2-3/мес
- Связь задач со стратегическими целями: % задач, явно привязанных к KR/гипотезам — тренд роста
- Доля решений, принятых командой без согласования CEO: тренд роста (baseline в первый месяц)
- Фреймворк «правильной гипотезы» формализован (FPF-ревью 2026-03-25): 3 критерия — связь с KR, измеримость (метрика + порог), срок проверки

## Product Scope

### Pre-MVP (Week -1 to 0)

**Контрольная точка (Day 1):**
- Тест транскрибации Soniox на наихудший сценарий: шумное помещение, быстрая речь, казахские вставки (code-switching русский↔казахский)
- Проверка Soniox API: async file transcription (загрузка аудиофайла → webhook с результатом). SDK: Python или Node.js
- Проверка потока: Google Meet запись → Google Drive → скачивание → Soniox API → Transcript Interface Contract
- Fallback: если API недоступен или запись не проходит порог качества → ручной экспорт/текстовая загрузка в бот (`/upload`)

**Артефакты Pre-MVP:**
- Инструкция трекера (подключение контрольный список, 2-3 стр.) — готовится к Day 1

**Подготовка Phase 1:**
- Подключение Geonline: F0 (стейкхолдерная карта + доменное знание) заполнена
- OKR и KR загружены в Google Sheets
- Промпты F1 и F4 протестированы на исторических транскриптах Geonline
- F5: определены 2 метрики (leading + lagging) для каждого департамента Geonline совместно с Дамиром. Запрошены источники метрик у клиента

**Начало кодификации (параллельно, для Phase 2):**
- Начало работы по методологическим требованиям (см. Domain-Specific Requirements): шляпы. Жизненный цикл гипотезы формализован (FPF-ревью 2026-03-25): 6 статусов с критериями переходов, см. `docs/tracking-process-as-is.md` §9.4

### MVP Phase 1 (недели 1-4): «Трекер + Клиент»

> Week 1 = разгон (первые 3 отчёта тестовые). Полный режим с Week 2. Детальная стратегия запуска — см. «Стратегия Scoping и приоритизация».

| Функция | Описание | Ценность |
|---------|----------|----------|
| F0 | Стейкхолдерная карта + доменное знание — Google Sheets шаблон, заполняется при подключении (~30 мин). Живой документ, ревизия ежемесячно | Контекст для AI |
| F1 | Автоотчёт по топу — транскрипт + контекст → structured output с цитатами → Telegram. Азиза проверяет, отправляет клиенту. **Двойная функция**: отчёт для клиента + инструмент accountability (зафиксированные обязательства с цитатами создают «цифровой след» обещаний) | Экономия 2ч/встречу + accountability |
| F4 | Подготовка к встречам — повестка в понедельник (3 пункта max): какие KR проверить, незакрытые action items, слепые зоны. **Ключевая feature для постановки практики**: повестка направляет трекера к вопросам, формирующим навык у команды (например: «спроси Жанель — какую гипотезу она хочет проверить и как измерит результат») | Трекер приходит подготовленной + формирует навык |
| F3-lite | Сводное уведомление для CEO — 5 строк + 🟢🟡🔴 по OKR + 1 инсайт/вопрос. Азиза проверяет перед отправкой. Понедельник утром | Прозрачность для CEO |
| F5 | Метрики департаментов — еженедельный сбор 2 ключевых метрик (leading + lagging) по каждому департаменту через Telegram-бот. Топ-менеджер отвечает 2 числами в 1 сообщение (~30 сек). Данные используются в F4 (повестка с объективными данными), F1 (верификация заявлений топа vs метрики), F3-lite (объективные 🟢🟡🔴 статусы). **Ключевая ценность**: pipeline видит реальную картину, а не только слова топов на сессиях | Объективность + раннее обнаружение проблем |

**Минимальная автоматизация:** Telegram Bot + Soniox API + Sheets API + Claude API.

**Поток ввода транскрипта (Вариант 3 — нативная запись):**
1. Азиза проводит сессию в Google Meet или Zoom и нажимает «Записать» (нативная функция платформы)
2. После сессии аудиофайл сохраняется в облако (Google Drive / Zoom Cloud)
3. Азиза отправляет боту ссылку на файл записи (один шаг)
4. Бот скачивает аудио, отправляет в Soniox API (async transcription), получает транскрипт через webhook
5. Транскрипт проходит через Transcript Interface Contract → F1 pipeline → отчёт

**Почему Soniox:** единственный из исследованных провайдеров (Soniox, Deepgram, MyMeet.ai, Timelist.ru), который поддерживает и русский (WER 6.2%), и казахский язык с автоматическим code-switching (переключение языков внутри реплики без конфигурации). Deepgram не поддерживает казахский в Nova-2/3. MyMeet.ai не поддерживает code-switching. Timelist.ru не поддерживает казахский. Цена Soniox ~$0.10/час — самая низкая.

**Fallback:** `/upload` — Азиза вставляет plain text транскрипт вручную (если Soniox API недоступен или запись не прошла порог качества).

### MVP Phase 2 (недели 3-6): «Руководитель + Полная система»

Показывается с оговоркой «AI-оценка» | Кодификация методологии |

## User Journeys

### Journey 1: Жанель (топ-менеджер клиента) — формирование навыка

**Opening Scene:** Жанель — РОП Geonline. Первая встреча с трекером (неделя 1). Азиза спрашивает: «Какие гипотезы по росту продаж ты тестируешь?» Жанель: «Ну... мы стараемся больше звонить.» Нет конкретики, нет метрик, нет срока.

**Rising Action:** Неделя 3. Азиза (с подсказкой F4): «Жанель, ты на прошлой неделе сказала что попробуешь видеозвонки. Какой результат? Какая конверсия?» Жанель: «О, да! Конверсия у Айгерим на видеозвонках 80%, у остальных 25%. Гипотеза подтверждается.» Впервые Жанель отчитывается в формате гипотеза → метрика → результат.

Жанель получает отчёт после встречи. Видит свои слова зафиксированными: «Жанель обязалась: перевести всех менеджеров на видеозвонки до 17 марта. Метрика: конверсия с 25% до 60%.» Accountability: это теперь записано, не забудешь.

**Climax:** Неделя 10. Азиза приходит на встречу. Жанель начинает сама: «Я хочу проверить новую гипотезу: внутренняя рассрочка для 10-х классов. Метрика: снижение неодобрений с 20% до 10%. Срок: 2 недели. Связано с KR-2.3.» Азиза не задавала вопрос — Жанель сформулировала сама, в правильном формате.

**Resolution:** Жанель научилась. Не потому что прочитала книгу — потому что 10 недель трекер задавал правильные вопросы, а отчёты фиксировали обязательства. Оргвозможность формируется через повторение: вопрос → ответ → фиксация → проверка → улучшение. Если прогресс не виден к неделе 6 — трекер и руководитель обсуждают: проблема в методе или в человеке?

### Journey 2: Азиза (трекер) — еженедельный цикл

**Opening Scene:** Понедельник, 9:00. Азиза открывает Telegram и видит повестку от бота на неделю. Три пункта по каждому из пяти топов Geonline. «Жанель: спроси про гипотезу с видеозвонками — результат должен быть готов. Мерей: кадровый контур не затрагивался 3 недели — уточни статус найма PM. Тұрар: KR по позиционированию не обсуждался 2 недели.»

Раньше Азиза тратила 30 минут на подготовку к каждой встрече — листала 20 листов Excel, вспоминала, что обсуждали. Теперь — 2 минуты на чтение повестки.

**Rising Action:** Вторник, 14:00. Встреча с Жанель (30 мин) в Google Meet. Азиза нажимает «Записать» в начале встречи. После встречи запись автоматически сохраняется в Google Drive. Азиза отправляет боту ссылку на файл. Один шаг. Бот скачивает аудио, транскрибирует через Soniox API (русский + казахские вставки распознаются автоматически). Через 10 минут — отчёт: решения, action items, статус гипотез, цитаты из разговора. Азиза читает, сверяет с памятью, поправляет одну формулировку, отправляет Жанель. Total: 30-40 минут (было 2 часа).

**Climax:** Пятница вечер. Все 5 встреч обработаны. Бот собрал сводное уведомление для Дамира. Азиза проверяет — всё верно, конфиденциального нет. Отправляет. Параллельно бот показывает: «На этой неделе ты 4 из 5 встреч больше давала советы, чем задавала вопросы. Попробуй на следующей неделе с Мереем начать с вопроса "что ты планируешь сделать по KR-7?"»

Азиза останавливается. Она и правда чувствовала, что с Мереем скатывается в экспертную позицию. Но раньше это было интуицией. Теперь — данные.

**Resolution:** Азиза потратила на рутину 1.5 часа за неделю вместо 4. Освободившееся время потратила на подготовку к сложному разговору с Арай про просроченные задачи — разговору, до которого раньше «руки не доходили».

### Journey 3: Марат (новый трекер) — онбординг

**Opening Scene:** Марат нанят как второй трекер ARB. Опыт: маркетолог, прошёл обучение методологии у Айдара. Первый клиент: новая компания после стратсессии. Марат получает: инструкцию трекера (2-3 страницы), доступ к Telegram-боту, шаблон стейкхолдерной карты.

**Rising Action:** Неделя 0. Марат заполняет стейкхолдерную карту (~30 мин), загружает OKR в Sheets. Неделя 1: первые встречи. AI не знает специфику нового клиента — отчёты generic. Марат дописывает доменное знание в Sheets. Отчёты улучшаются.

**Climax:** Неделя 3. F4 присылает повестку: «Спроси финдиректора — какую гипотезу по оптимизации расходов он хочет проверить?» Марат — маркетолог, не финансист. Без повестки он бы спросил про маркетинг. Повестка вытащила его из слепой зоны. Параллельно Айдар получает QC: talk ratio Марата 55/45 — говорит чуть больше нормы. Айдар: «Больше слушай, меньше советуй.»

**Resolution:** Неделя 6. Качество отчётов и подготовки к встречам Марата — на уровне Азизы. Качество живых сессий — растёт быстрее, чем было бы без AI, но ещё не на уровне опытного трекера. AI компенсирует недостаток опыта в структуре, но не заменяет мастерство живой работы с людьми.

### Journey 4: Дамир (CEO-клиент) — постановка практики

**Opening Scene:** Понедельник, 8:30. Дамир открывает Telegram. Сводное уведомление от трекера:

🟢 OKR-3 (SuperApp): +2700 посещений, гипотеза с учениками в тесте
🔴 OKR-2 (Продажи): конверсия 25%, третья неделя ниже плана
🟡 OKR-5 (Маркетинг): темп гипотез недостаточен
💡 Инсайт: команда за месяц сама инициировала 4 гипотезы (было 1)

30 секунд чтения.

**Rising Action:** Месяц 1 → 3. Дамир привыкает к ритму. Замечает тренд: команда каждую неделю ставит задачи, формулирует гипотезы. Не потому что Дамир попросил — потому что трекер научил.

**Edge case:** Неделя 4: Дамир не открывал уведомление 2 недели. Азиза спрашивает на встрече: «Видели уведомление?» — «Нет, некогда.» Азиза меняет формат: вместо 5 строк — одно предложение + один вопрос. Дамир начал реагировать.

**Climax:** Месяц 5. Уведомление показывает: команда за 4 недели самостоятельно запустила 6 гипотез, 3 дали результат. Жанель впервые пришла на встречу с готовой гипотезой и метрикой — без подсказки трекера. Решения всё ещё согласовываются с Дамиром, но инициатива — от команды. Прогресс: от 0 самостоятельных инициатив к 4 за месяц.

**Resolution:** Дамир продлевает контракт. Не потому что привык к дашборду — потому что видит: трекинг реально меняет его команду, и данные это подтверждают.

### Journey 5: Айдар (руководитель ARB) — QC-контроль

**Opening Scene:** Пятница, 17:00. У Айдара 3 клиента, 2 трекера (Азиза и Марат). Раньше раз в 2 недели прослушивал записи — выборочно. Сейчас открывает Telegram — QC-сводка за неделю.

**Rising Action:** Сводка: Азиза — стабильно. Марат — алерт: talk ratio 60/40, покрытие OKR 60%, 70% утверждений vs 30% вопросов. Марат скатывается в экспертную позицию. Клиент пока не жаловался, но тренд опасный.

**Climax:** Айдар звонит Марату. «Посмотри свои цифры. 70% утверждений. С Асель попробуй начать с открытого вопроса и промолчать 30 секунд после ответа.» Конкретная рекомендация с данными, не абстрактное «старайся больше слушать».

**Resolution:** Через 2 недели talk ratio Марата — 40/60. Клиент не пожаловался. Айдар предотвратил проблему, не выходя из Telegram.

### Journey 6: Тимур (ops/developer) — поддержка pipeline

**Opening Scene:** Среда, 22:00. Telegram: «⚠️ Bot error: Claude API timeout. Retry 1/3 failed. Запрос от Азизы (встреча с Тұрар).» Тимур проверяет логи — третий retry прошёл, отчёт сгенерирован с задержкой 25 минут. Азиза не заметила.

**Rising Action:** Пятница — агрегация F3-lite не сработала: Sheets API токен истёк. 15 минут на обновление, перезапуск. Уведомление Дамиру ушёл с задержкой 2 часа. Тимур добавляет автообновление токена в cron.

**Edge case (тихая деградация):** Неделя 4: Азиза замечает, что отчёты стали менее конкретными — «какие-то общие фразы». Тимур проверяет: Claude обновил модель, промпт ведёт себя иначе. 2 часа на адаптацию промпта + тест на 5 исторических транскриптах. Инсайт: нужен regression test при обновлении модели.

**Resolution:** Время на ops за неделю: 45 минут (+ 2 часа разово на промпт). В пределах бюджета. Система работает, но требует внимания — не «fire and forget».

## Domain-Specific Requirements

### Конфиденциальность и согласие

- **Согласие на запись (opt-in):** Каждый участник встречи должен дать согласие на запись, транскрибацию и AI-обработку. Формальный документ — от юриста. Прагматичный подход: устное согласие на первой встрече (Day 1), формальное подписание к Week 3-4
- **Информирование:** Азиза на первой встрече с каждым топом: «Встречи записываются для создания отчётов. AI обрабатывает транскрипт. Вы согласны?» Момент фиксируется в транскрипте Soniox
- **Отказ от записи:** Если топ не согласен — встреча без записи, отчёт вручную. Pipeline пропускает эту встречу
- **Конфиденциальность высказываний:** Трекер не передаёт CEO конфиденциальные высказывания сотрудников. Уведомление CEO никогда не содержит прямых цитат топов — только агрегированные наблюдения
- **Оценка чувствительности:** Задача трекера (human judgment), не AI. Трекер при проверке отчёта оценивает чувствительность контента и удаляет конфиденциальное перед отправкой клиенту

### Data Retention

- **Хранение:** Срок действия договора + 12 месяцев. После — обезличивание или удаление по запросу клиента
- **Доступ:** Тимур — полный. Трекер — данные своих клиентов. Айдар — QC-сводки. CEO-клиент — только уведомление и отчёты через трекера
- **Архитектура (Growth):** Разделение raw data (транскрипты с именами, удаляемо по запросу) и aggregated data (тренды, метрики без привязки к персоне, остаётся)
- **Policy:** Полная data retention policy — формализовать к Phase 2

### Методологические требования (каноническое место)

**Для Phase 1 (Тимур формулирует, Айдар утверждает):**
- **Жизненный цикл гипотезы:** ✅ Формализовано (FPF-ревью 2026-03-25): 6 статусов с критериями переходов, см. `docs/tracking-process-as-is.md` §9.4
- **Базовое определение правильной гипотезы (3 критерия):** ✅ Формализовано (FPF-ревью): связь с KR, измеримость (метрика + порог), срок проверки
- **Внешние бизнес-метрики (F5):** 2 метрики на департамент (leading + lagging). Определяются совместно с CEO при подключении клиента. Типовые: продажи (лиды + выручка), маркетинг (кол-во лидов + CAC), HR (eNPS + текучесть), финансы (cash flow + маржинальность), продукт (DAU + NPS). См. `research/methodology-and-metrics-research-2026-03-27.md` §2.2

**Для Phase 2 (исследование проведено 2026-03-27, нужно утверждение Айдара):**
- **Кодификация шляп трекера:** Предложена модель 4 шляп: коуч (35-45%), эксперт (15-25%), трекер (20-30%), фасилитатор (10-20%). Fallback на 3 (объединить фасилитатор + трекер) если inter-rater agreement < 70%. Обоснование: анализ практики Азизы + Gong + ICF frameworks. См. `research/methodology-and-metrics-research-2026-03-27.md` §1.1
- **Пороги talk ratio:** Talk ratio трекера: норма 30-45%, жёлтый алерт 45-55%, красный > 55%. Монолог трекера ≤ 2 мин. Основано на Gong (43:57 для sales) и ICF (20:80 для чистого коучинга) — ARB между ними. Контекстные исключения: первая встреча, кризис, ранняя стадия обучения. См. `research/methodology-and-metrics-research-2026-03-27.md` §1.2
- **Покрытие OKR:** ≥ 2 содержательных реплики = «обсуждено» (факты, метрики или решения). 1 реплика = «упомянуто». Слепая зона = пропущено 2+ нед подряд → F4 алерт. Целевое покрытие: 100% KR за 2 недели. См. `research/methodology-and-metrics-research-2026-03-27.md` §1.3

**Для Growth (исследовательские задачи):**
- **Стратегическая верность гипотезы:** Связь с OKR, бьёт в главное ограничение — расширение определения за рамки формальной корректности. Доменное исследование + подбор/адаптация фреймворка
- **Педагогический подход:** Как трекер формирует у команды навык постановки гипотез — устойчивое мастерство, не зависимость от трекера. Педагогический цикл: вопрос → ответ → фиксация → проверка → улучшение
- **Оргвозможность как измеримый результат:** 8 компонентов, из которых AI наблюдает 4-5 (постановка задач, формулировка гипотез, связь с целями, перепланирование). Выполнение обязательств и самостоятельность решений — между встречами, вне зоны AI. Главная метрика — скорость обучения (тренд), не абсолют
- **Accountability через цифровой след:** F1 фиксирует обязательства с цитатами — мягкое социальное давление, формирующее привычку выполнять обещания

### Доменные ограничения

- **Адаптация под отрасль клиента:** Первый клиент в новой отрасли требует 1-2 недели на перенастройку промптов (терминология, роли, OKR-контекст). Учитывать в графике подключения
- **Методологическая валидность:** AI-оценки (шляпы, talk ratio, покрытие OKR) показываются с оговорка «AI-оценка», не как абсолютная истина
- **Язык AI-output:** Русский. Казахскоязычные цитаты из транскрипта — в оригинале с пометкой

## AI Automation Pipeline — Технические требования

> *Секции 1-6 — технические детали для разработчика и архитектора. Бизнес-стейкхолдерам рекомендуется начать с «Фазы глазами ролей» в конце секции.*

### 1. Архитектура Pipeline

#### Обзор

Система — AI automation pipeline с push-first моделью доставки и контроль трекера (трекер проверяет и одобряет каждый output перед отправкой клиенту). MVP — линейный конвейер с тремя pipeline: F1 (автоотчёт, real-time), F4 (повестка, batch), F3-lite (сводное уведомление, batch).

**Эволюция типа проекта:**
- **MVP (Phase 1-2):** AI Automation Pipeline — линейный конвейер с Telegram-ботом
- **Growth (v2.0):** Pipeline + веб-дашборд
- **Target (v3.0):** AI-Powered Consulting QA & Coaching Platform — мультиагентная архитектура

#### Transcript Interface Contract

Формализованный вход pipeline — JSON schema, провайдер-агностичный:

```json
{
  "speakers": [{"name": "string", "segments": [{"start": "float", "end": "float", "text": "string"}]}],
  "metadata": {"date": "string", "duration": "float", "meeting_type": "string"}
}
```

Парсер Soniox и plain-text fallback — отдельные модули, конвертирующие вход в единый contract. Смена транскриптора не требует переписывания pipeline.

#### Input Validation (после парсинга)

- Все speakers mapped (нет unnamed segments)
- Нет пустых сегментов
- Timestamps монотонно возрастают
- Speaker sanity check: если > 70% реплик speaker'а не соответствуют ни одной из его ролей по стейкхолдерной карте — пометить [speaker_check] для ручной проверки трекером (порог 70% отсекает ложные срабатывания при мультиролевости)
- Если validation failed → уведомление Тимуру, pipeline останавливается до разбора

#### Chain of Prompts (ADR: почему chain, а не один промпт)

Транскрипт 30 мин = 10-15K слов. С OKR-контекстом и стейкхолдерной картой — под 50K токенов. Один промпт — хрупкость: нельзя дебажить по частям, один сбой = всё переделывать. Chain позволяет верифицировать каждый этап отдельно.

| Этап | Вход | Выход |
|------|------|-------|
| 1. Extraction | Транскрипт + OKR + стейкхолдерная карта | Факты, решения, цитаты с timestamps/speakers, **commitments** `[{who, what, deadline, quote}]` (accountability engine) |
| 2. Analysis | Extraction results + OKR данные | Покрытие OKR, статус гипотез, [speaker_check] метки |
| 3. Formatting | Analysis results | Отчёт по шаблону роли |
| 4. Delivery | Formatted output | Telegram/Email сообщение |

Встроенная citation verification в extraction: промпт требует точный timestamp для каждой цитаты. Если точное совпадение не найдено — помечается [approximate]. Отдельный verification step — если Азиза в первые 2 недели найдёт > 2 ложных цитат (не строим защиту до подтверждения проблемы).

#### Graceful Degradation

Если step 3 (analysis) или step 4 (formatting) failed после 3 retries — отправить Азизе partial result: «Извлечённые данные (без форматирования). Formatting не удался, Тимур уведомлён.» Азиза получает сырые данные и пишет отчёт быстрее, чем с нуля (30 мин вместо 2 часов).

#### Три Pipeline

**F1 — Автоотчёт (Real-time, после встречи):**
- Trigger: `/report <url>`
- Input: транскрипт + OKR + стейкхолдерная карта
- Chain: extraction → analysis → formatting → delivery
- SLA: < 15 мин (latency budget: 4 steps × 2-3 мин = 8-12 мин)
- Fallback: partial results при сбое step 3-4

**F4 — Повестка (Batch, понедельник):**
- Trigger: cron, понедельник 9:00 (после F5 — метрики уже собраны)
- Input: все F1 за неделю + OKR + прошлые action items + commitments + **F5 метрики департаментов**
- Output: повестка по каждому топу (3 пункта max): какие KR проверить, незакрытые commitments, слепые зоны, **расхождения метрик vs заявлений** (например: «Жанель сказала конверсия растёт, но F5 показывает снижение с 25% до 20% — уточни»)
- SLA: готово к 9:30
- Fallback: трекер готовит вручную

**F5 — Метрики департаментов (Batch, понедельник утро):**
- Trigger: cron, понедельник 8:00 (до F4)
- Input: бот отправляет каждому топ-менеджеру запрос на 2 числа (leading + lagging метрика)
- Collection: топ отвечает числами в Telegram (~30 сек), данные записываются в Sheets
- Output: тренды по департаментам + алерты на аномалии (значительное отклонение от тренда)
- Consumers: F4 (повестка с объективными данными), F1 (верификация: слова vs метрики), F3-lite (объективные 🟢🟡🔴)
- SLA: данные собраны до генерации F4 (до 9:00). Если топ не ответил — F4 генерируется без свежих метрик, используются прошлонедельные
- Fallback: если бот не работает — Google Form ссылка

**F3-lite — Сводное уведомление (Batch, понедельник):**
- Trigger: после F4 или параллельно
- Input: все F1 за неделю + F5 (метрики департаментов)
- Output: 5 строк + 🟢🟡🔴 по OKR + 1 инсайт/вопрос к CEO. 🟢🟡🔴 статусы теперь подкреплены объективными метриками F5, а не только словами топов
- Delivery: Telegram/email **от имени трекера** (pipeline невидим для клиента)
- Gate: Азиза approve перед отправкой CEO
- Fallback: трекер пишет уведомление вручную

### 2. Интеграции

#### Транскриптор

- Основной: Soniox API (приоритет по качеству транскрибации и code-switching русский/казахский)
- Альтернатива: tldv или иной источник экспорта транскрипта
- Fallback: ручной экспорт + загрузка в бота
- Решение: Soniox принят как основной транскрибатор. Контрольная точка Day 1 подтверждает качество и устойчивость API, а не выбирает между провайдерами
- Контрольная точка включает тест на шумной записи и code-switching

#### Google Sheets

- OKR-трекер, стейкхолдерная карта — input для pipeline
- Data access adapter: pipeline читает/пишет через абстракцию (один модуль, ~50 строк), не напрямую Sheets API. Замена хранилища = замена одного модуля (потому что Sheets API имеет quirks: merged cells, формулы, encoding, rate limits 100 req/100sec)
- Auto-refresh API tokens с Day 1 (потому что silent token expiry → каскадный отказ, нет уведомление'а)
- При масштабировании на 3+ клиентов: миграция на выделенное хранилище

#### Telegram Bot — Interaction Interface

**Команды и действия трекера:**

| Механизм | Действие |
|---------|----------|
| `/report <url>` | Запуск генерации отчёта. Бот подтверждает: «Отчёт будет через ~10-15 мин» |
| Inline-кнопка `[✅ Подтвердить → {Name}]` | Подтвердить текст и перевести в статус `approved` |
| Inline-кнопка `[✏️ Исправить]` + reply | Точечная правка текста перед отправкой |
| Inline-кнопка `[❌ Отклонить]` | Отклонить и перегенерировать/уйти в ручной режим |
| Bot Menu `[📊 Статус]` | Посмотреть очередь и состояние pipeline |

**Сбор метрик от топ-менеджеров (F5):**

Бот отправляет каждому топу еженедельный запрос (Пн, 8:00) в формате:

```
📊 Метрики за прошлую неделю:
• [leading метрика]: ___
• [lagging метрика]: ___
Ответь двумя числами через запятую
```

Топ отвечает: `150, 2500000` — бот записывает в Sheets. Если топ не ответил до 9:00 — повторный запрос. Если не ответил до 12:00 — F4/F3-lite используют прошлонедельные данные, трекеру приходит пометка «метрики [имя] не обновлены».

Формат доставки отчёта трекеру: Telegram message (до 4096 символов). Если отчёт длиннее — разбивка на 2-3 сообщения с заголовками секций. Альтернатива (по запросу Азизы): ссылка на Google Doc.

Reject > 50% отчётов за неделю → пересмотр промптов (потому что если больше половины отчётов reject — AI не помогает, а мешает — сигнал остановки).

#### Email

- CEO-клиент: сводное уведомление F3-lite в понедельник (дополнительный канал к Telegram)
- Email emergency mode для трекера — по запросу, если Азиза столкнётся с проблемой доступа к Telegram. Fallback protocol покрывает этот кейс без отдельной разработки

### 3. Модель данных

#### Изоляция клиентов

- Каждый клиент — полностью изолированный контекст. Данные одного клиента никогда не попадают в промпты другого
- Промежуточные данные chain хранятся в папках по client_id
- Все данные клиента хранятся под одним client_id — deletion = удаление по ID (принцип для Growth: data map + контрольный список deletion при offboarding)

#### Разделение доступа

- MVP: логическое — на уровне промптов и output-шаблонов. Один Telegram-бот, разные сообщения разным ролям
- При подключении 2-го клиента: обязательный code review на изоляцию контекстов (client_id isolation review)
- При 2+ трекерах: отдельные чаты в Telegram или thread-based isolation
- Жёсткое техническое разделение — при 5+ клиентах

#### Невидимость pipeline для клиента

- Уведомление CEO отправляется от имени трекера, не от бота
- CEO общается с трекером напрямую (WhatsApp/звонок), не через pipeline
- Отсутствие трекера = pipeline стоит — контроль трекера обязателен, by design

#### Метрики департаментов (F5)

- Отдельный Sheet/таблица в OKR-трекере: `client_id`, `department`, `metric_name`, `metric_type` (leading/lagging), `value`, `week`, `source` (manual_telegram/manual_form/api), `reported_by`
- Phase 1: ручной ввод через Telegram-бот
- Growth: автоматический сбор через API (AmoCRM, Bitrix24, 1C, Google Analytics) — data access adapter расширяется, не переписывается
- При подключении нового клиента: аналитик практики определяет 2 метрики × N департаментов совместно с CEO, заносит в F0 (стейкхолдерная карта)

#### QC-метрики

- Phase 1: ops-логи достаточно (QC-скоринг ещё не реализован)
- Phase 2: append-only лог QC-метрик (отдельный Sheet) для трендового анализа за 3-6 месяцев. Архитектурно заложить в data adapter (append-only метод), реализовать когда данные появятся

### 4. Операционная модель

#### Ops Logging (zero-cost stack)

- Логи каждого step chain: append-only Sheet (отдельный от данных клиента)
- Дублирование критических событий в Telegram-канал ops (независимо от Sheets — потому что если Sheets упал, логи туда тоже не пишутся, каскадный отказ)
- Три строки на каждый run: pipeline started → steps completed → result delivered

#### Monitoring и Уведомлениеing

| Событие | Действие | Кто получает |
|---------|----------|--------------|
| Отчёт не готов за 30 мин | Уведомление: «Переключайся на ручной режим, напиши Тимуру» | Азиза |
| Pipeline error (любой step) | Уведомление с деталями ошибки | Тимур |
| Pipeline down > 4ч | Повторный уведомление | Тимур |
| Pipeline down > 24ч | Тимур пишет вручную | Айдар |
| Уведомление CEO не отправлен в понедельник | Трекер отправляет ручной уведомление или предупреждение | Дамир |

#### Weekly Canary Test

- Каждый понедельник pipeline прогоняет один транскрипт из golden dataset и сравнивает с эталоном
- Golden dataset: 5 исторических транскриптов Geonline с эталонными отчётами, размеченными Азизой pre-MVP (5 — минимум для покрытия разных типов встреч: маркетинг, HR, финансы, продажи, общая синхронизация)
- Canary включает проверку формата входных данных, не только quality output
- Diff > 30% по структуре или пропуск ≥ 2 ключевых фактов → уведомление Тимуру
- Cost: ~$0.50/нед

#### Prompt Versioning

- Промпты хранятся в git, версионируются
- Regression threshold: структурный diff > 30% или пропуск ≥ 2 ключевых фактов

#### Fallback Protocol (для трекера)

- Отчёт не пришёл за 30 мин → переключиться на ручной режим
- Написать Тимуру в Telegram
- Не ждать починки, не пытаться resend — продолжить работу

#### Ops-бюджет Тимура

| Задача | Время/нед (штатно) |
|--------|-------------------|
| Canary test review | 15 мин |
| Уведомление handling | 15 мин |
| Ad-hoc (промпт-тюнинг, дебаг) | 30 мин |
| **Итого** | **~1ч/нед** |

При нестабильном pipeline — до 2ч/нед (бюджет из Success Criteria). Если > 2ч/нед стабильно — сигнал к пересмотру архитектуры.

## Стратегия Scoping и приоритизация

### MVP-философия

**Problem-solving MVP:** доказывает три preconditions, необходимых для инвестирования в постановку практики:

1. **AI корректно структурирует данные** из трекинговых сессий (extraction quality)
2. **Трекер использует AI-output** в своей работе и считает его полезным (принятие)
3. **Механизмы accountability и подготовленности работают** (behavioral change signal)

Постановка практики стратегического исполнения — стратегический горизонт post-Phase 2. На MVP (6 недель) невозможно измерить тренд качества гипотез — слишком мало данных. Но если preconditions подтверждены — есть основания инвестировать дальше.

**Ключевое допущение:** AI-отчёт достаточно хорош, чтобы Азиза доверяла ему и отправляла клиенту, а не переписывала с нуля. Если не подтвердится — весь pipeline бесполезен.

### Функции Day 1 и их роль

**F4 (повестка) — ключевая возможность для постановки практики.** Направляющие вопросы формируют навык у команды: «спроси Жанель — какую гипотезу она хочет проверить и как измерит результат». F1 без F4 = экономия времени, но не постановка практики. F4 без F1 = невозможна (нет данных). F1+F4 неразделимы.

| Функция | Роль в постановке практики |
|---------|----------------------|
| **F0** | Контекст — без него AI генерирует плоские отчёты |
| **F1** | Accountability — цифровой след обязательств (commitments с цитатами) |
| **F5** | Объективность — реальные метрики vs слова. Раннее обнаружение проблем до сессии |
| **F4** ★ | Подготовленность — направляющие вопросы формируют навык у команды + верификация через F5. **Ключевая возможность** |
| **F3-lite** | Страховка бизнес-модели — CEO видит прогресс с объективными метриками, продолжает платить |

### Приоритизация при ресурсном давлении

Если промпты не стабилизируются (R4: reject > 50% за 2 последовательные недели) и ops-время вырастет до 5-6ч/нед:

**Порядок degradation:**
1. **Отложить F3-lite** — Азиза отправляет ручной уведомление CEO, Дамир не заметит разницы
2. Фокус на стабилизации **F1 + F4** — они ставят практику
3. F3-lite возвращается после стабилизации (1-2 недели)

**Не откладываем при ресурсном давлении:** F1 (ядро pipeline), F4 (ключевая возможность), Контроль трекера (доверие к системе). При сигнал остановки — отдельное решение.

### Must-Have (без этого продукт не работает)

- Транскрибация с приемлемым качеством (критерий остановки)
- Structured output с цитатами-источниками (защита от галлюцинаций)
- Commitments extraction `[{who, what, deadline, quote}]` (accountability engine)
- Повестка с направляющими вопросами (формирование навыка — ключевая возможность)
- Контроль трекера (доверие к системе → принятие)
- Fallback protocol (трекер не зависает при сбое)
- Метки [approximate] и [speaker_check] в отчёте — трекер проверяет помеченное, не весь текст (механизм принятия, обучающий механизм для калибровки доверия в первые 2-3 недели)
- Graceful degradation — partial results при сбое step 3-4 (30 мин разработки, значительно улучшает reliability)

### Nice-to-Have Phase 1

- Email как дополнительный канал для CEO
- Provider change detection (hash структуры ответа)
- Email emergency mode для трекера

### Отложено до Phase 2 (зависит от данных)

- QC-скоринг F2 (нужна кодификация методологии)
- Фидбэк трекеру по шляпам (нужны данные за 3+ недель)
- Append-only QC лог для трендов
- Context drift detection

### Отложено до Growth (зависит от масштаба)

- Веб-дашборд, мультиагентная архитектура, RAG-база
- Пересмотр approve workflow при 3+ клиентах на трекера (контроль трекера узкое место: 10 встреч/нед × 10-15 мин approve = 2+ч/нед только на проверку)
- Подключение новых трекеров с AI

### Pre-MVP (1-2 дня, два трека)

> Состав задач Pre-MVP — см. «Product Scope → Pre-MVP». Здесь — порядок и логика запуска.

**Контрольная точка — первая задача Тимура.** Если транскрибация неприемлема — стоп. Азиза начинает только после GO. Два параллельных трека:

- **Трек Тимура (1-2 дня):** контрольная точка → тест промптов → деплой
- **Трек Азизы (после GO, 2-3 часа):** F0 (30 мин) + разметка 3 эталонных отчётов для golden dataset (3 типа: цифровая встреча, качественная, code-switching; остальные 2 добираются в Week 1) + устное согласие

### Подключение Азизы

- Инструкция-картинка (скриншот бота, 3 шага): отправь ссылку → подожди 10-15 мин → approve/edit/reject
- Первый отчёт Тимур генерирует вместе с Азизой в чате
- Первые 3 отчёта — тестовые, не отправляем клиенту (покрытие типов встреч: разные топы, разные контуры; при 5 встречах/нед = 2-3 дня, не неделя)
- Метки [approximate]/[speaker_check] — обучающий механизм для первых 2-3 недель. Азиза проверяет помеченное, не весь текст. По мере калибровки доверия — перестаёт использовать. Это нормально
- Еженедельная анкета (уже в Success Criteria) + один вопрос: «сколько примерно минут тратишь на проверку одного отчёта?»

| Период | Что | Ключевая метрика |
|--------|-----|-----------------|

### Growth: что учесть при масштабировании

- **Каждый новый клиент = 1-2 недели адаптации промптов** (другая отрасль, терминология, структура OKR). Это повторяемая задача, не one-off
- **Контроль трекера узкое место при 3+ клиентах на трекера:** 10+ approve/нед × 10-15 мин = 2+ч/нед. Рассмотреть batch approve или auto-approve для отчётов с 0 метками
- **F4 — конкурентное преимущество:** новый трекер с F4 выходит на уровень опытного за 3 недели вместо 3 месяцев. Это selling point при найме трекеров и продаже услуги

## Functional Requirements

> **Это контракт capability.** UX-дизайнер проектирует только то, что здесь. Архитектор поддерживает только это. Эпики реализуют только это. Если capability нет в этом списке — её не будет в продукте.

### 1. Обработка трекинговых сессий

- **FR1** [MVP]: Трекер может отправить ссылку на запись встречи и получить структурированный отчёт

- **FR12** [MVP]: Система может сгенерировать компактный сводное уведомление с оценкой статуса каждого OKR и ключевым инсайтом для CEO

## Non-Functional Requirements

> NFR определяют **как хорошо** система работает, не что она делает. Каждый NFR — measurable quality attribute.

### Performance

- **NFR1** [MVP]: Генерация отчёта F1 (от получения ссылки до доставки трекеру) — цель < 15 мин, уведомление при > 30 мин
- **NFR2** [MVP]: Генерация повестки F4 (batch, понедельник) — готово к 9:00 при запуске cron в 7:00
- **NFR3** [MVP]: Генерация F3-lite сводное уведомление — готово к 9:00 (параллельно с F4 или после)
- **NFR4** [MVP]: Время ответа Telegram бота на команду (acknowledge) — < 3 сек
- **NFR5** [Phase 2]: Генерация QC-скоринга F2 — в течение 1 часа после обработки всех встреч за неделю

### Security & Конфиденциальность

- **NFR6** [MVP]: Изоляция клиентов верифицируема: данные клиента A не могут быть получены через контекст клиента B ни на одном этапе pipeline. Проверяется code review при подключении 2-го клиента
- **NFR7** [MVP]: Данные pipeline не используются для обучения AI моделей (подтверждается data policy провайдера — Anthropic). Принятый риск: серверы за пределами РК
- **NFR8** [Growth]: Полное удаление данных клиента занимает < 1 часа и подтверждается контрольный список (data map по client_id)

### Reliability

- **NFR9** [MVP]: Pipeline reliability: ≤ 2 неустранимых сбоя за 3 недели (метрика с Week 2, Week 1 = разгон)
- **NFR10** [MVP]: Временные сбои внешних API не приводят к потере данных или пропуску отчёта
- **NFR11** [MVP]: При сбое этапов analysis/formatting — доставка partial results (extraction data) трекеру вместо полного отказа
- **NFR12** [MVP]: Сбой Google Sheets не приводит к потере логов и алертов pipeline
- **NFR13** [MVP]: Истечение API tokens не приводит к простою pipeline без уведомление'а

### Integration

- **NFR14** [MVP]: Смена провайдера транскрипции требует только создания нового парсера, без изменений в pipeline логике
- **NFR15** [MVP]: Замена хранилища данных не требует изменений в бизнес-логике pipeline

### Scalability

- **NFR16** [MVP]: Telegram бот доступен 24/7 (uptime > 99% за месяц)
- **NFR17** [MVP]: Система поддерживает 1 клиента, 1 трекера, ~5 встреч/нед, ~20-25 API calls/нед
- **NFR18** [Growth]: Система поддерживает 5-10 клиентов, 2-3 трекера, ~50 встреч/нед без изменения архитектуры pipeline
- **NFR19** [Growth]: Approve workflow не занимает > 3ч/нед у трекера при 3+ клиентах (пересмотр при превышении)

### Cost & Operations

- **NFR20** [MVP]: Стоимость API pipeline — < $100/мес при штатной нагрузке (5 встреч/нед)
- **NFR21** [MVP]: Качество транскрибации: WER < 15% на наихудший сценарий (шум, быстрая речь, code-switching) — критерий остановки
- **NFR22** [MVP]: Ops-время ≤ 2ч/нед при стабильном pipeline (Week 2+). Week 1: до 3-5ч (разгон)

# Дизайн-записка Story 8.5: два способа завести клиента — импорт готового Excel vs сборка из документов

Дата: 2026-07-08 · Статус: **УТВЕРЖДЕНО ТИМУРОМ 2026-07-09**. Ответы на §6:
1) фикстуры — только Geonline; 2) текст через сборку — ок; 3) автодетект + кнопки — ок;
4) после импорта добавить кнопку «🧠 Досинтезировать гипотезы» (вошло в объём 8.5).

**Проблема, которую решаем:** сегодня бот принимает md/txt/docx/pdf и «собирает» стратегию
через LLM. Но у части клиентов стратегия уже готова и лежит в Excel — а `.xlsx` бот
отвергает как неподдерживаемый файл (это находка W11 из прод-фидбэка, требование FR102).

## 1. Главное решение: импорт встраивается в существующий конвейер, а не строит новый

Как онбординг работает сейчас, по шагам:

1. Азиза кидает документы → LLM вычитывает их и составляет **черновик стратегии**
   (цели, KR, участники, гипотезы) в едином внутреннем формате.
2. Дальше всё работает с этим черновиком: бот показывает саммари, подсвечивает
   дыры (🔴 нет базы/цели/ответственного), задаёт вопросы дозаполнения
   (диалог 7.3 + улучшения 8.6), и после `/confirm` выгружает в Google Sheets.

**Предложение:** импорт Excel — это просто второй способ получить тот же черновик.
Вместо «LLM читает документы» будет «код читает таблицу» (ноль обращений к LLM —
быстро, бесплатно, ничего не выдумывает). Всё, что после черновика — саммари,
проверки, вопросы, выгрузка — переиспользуется как есть, без единой правки:

```
Путь «сборка»: файлы (md/txt/docx/pdf) → LLM → черновик ┐
Путь «импорт»: один .xlsx → код (без LLM) ──────────────┤→ дальше одинаково:
                                                        │  саммари → вопросы про
                                                        │  дыры → /confirm → Sheets
```

Что это даёт на практике:

- Неполные строки из Excel (KR без базы или ответственного) не ломают импорт —
  они штатно попадают в 🔴 и в вопросы дозаполнения, как будто их «недособрал» LLM.
- Все существующие проверки и предупреждения работают на импортированных данных
  автоматически.
- Нового состояния почти нет: в сессию добавляется одно необязательное поле «каким
  путём идём» — старые сессии остаются совместимыми.

## 2. Как Азиза выбирает путь

- При `/newclient` к приветствию добавляются две кнопки:
  **«📥 Есть готовая стратегия (Excel)»** и **«🧠 Собрать из документов»**.
- Нажимать кнопку **не обязательно** — бот сам поймёт по первому файлу:
  прислали `.xlsx` → импорт; прислали md/txt/docx/pdf → сборка как сейчас.
  То есть для привычного сценария ничего не меняется, лишних кликов нет.
- Смешивать пути в одной сессии нельзя (иначе непонятно, что с чем объединять):
  - в режиме импорта прислали второй файл или документ → бот подскажет: «импорт
    работает по одному Excel; для пакета документов начни заново через /newclient»;
  - в режиме сборки прислали `.xlsx` → бот предложит кнопку «переключиться на импорт»
    (с подтверждением, если уже накоплены другие файлы).
- Если импорт не получился (см. §3, «когда импорт отказывает») — бот честно объясняет
  почему и предлагает пойти через сборку из документов.

## 3. Как код читает Excel

Новый модуль `src/f0-import.ts`. Библиотека для чтения xlsx уже есть в проекте
(мы ей уже читали таблицу Geonline скриптом), так что новых зависимостей нет.

**Случай А — «наша» таблица** (файл, выгруженный из нашего же шаблона трекинга,
как у Geonline). Узнаём её по служебному листу `_okr` или по знакомым заголовкам
колонок. Тут соответствие колонок известно заранее: «key_result» → формулировка KR,
«current_status» → база, «target» → цель, «owner» → ответственный и т.д. KR
группируются в цели по колонке «okr_group», лист участников → участники, лист
гипотез (если есть) → гипотезы.

**Случай Б — произвольная таблица клиента.** Бот ищет лист, похожий на таблицу KR:
считает, сколько заголовков колонок он узнал по словарю синонимов («KR / ключевой
результат / результат», «база / текущее / с», «цель / до / план», «ответственный /
владелец / кто», «срок / дедлайн / когда», «направление / цель года»). Узнал 3 и
больше — лист берём. Цели строим по колонке «направление» (нет такой — вся таблица
идёт одной целью с названием листа). Участники — уникальные имена из колонки
«ответственный». Гипотезы **не выдумываем**: если в файле нет явного блока гипотез,
в черновике будет честный «Гипотез: 0» — Азиза добавит руками или прогонит сборку.

**Что не распозналось — показываем, а не прячем.** Колонки, которые не удалось
сопоставить, и пропущенные листы попадают в блок «Не распознано» черновика — строками
вида «лист „Стратегия“: колонки „Вес“, „Комментарий“». Пустые ячейки ничем не
заполняем: пустая база → 🔴 → вопрос дозаполнения (уже с проверкой чисел из 8.6).

**Когда импорт отказывает:** ни один лист не похож на таблицу KR, ноль распознанных
строк, файл битый или слишком большой. Во всех случаях — понятное сообщение с причиной
и предложение пути «сборка из документов». Никакого «молча получился пустой черновик».

Мелкая деталь: все KR из Excel считаем метриками (не вехами). Пытаться угадывать
вехи по таблице — гадание; цена ошибки — максимум один лишний вопрос дозаполнения.

## 4. Что сознательно НЕ делаем в этой story (и почему)

1. **Не подключаем LLM для угадывания колонок.** Если словарь синонимов не справился
   с чьей-то экзотической таблицей — есть запасной путь (сборка из документов, там LLM
   и так работает). Если реальные файлы клиентов покажут, что словаря мало, добавим
   один дешёвый LLM-вызов «сопоставь заголовки» отдельной задачей.
2. **Не пишем отдельный парсер «стратегии текстом».** Готовая стратегия, присланная
   текстом или в docx, отлично проходит через существующую сборку — писать для неё
   свой разборщик таблиц хрупко и дублирует LLM.
3. **Не принимаем несколько xlsx или xlsx вперемешку с документами** — один Excel
   на импорт.
4. **Не поддерживаем старый `.xls` и `.csv`** — вне объёма MVP (csv легко добавить
   позже, если попросят).

## 5. Объём работ

- `src/f0-import.ts` (новый, ~250 строк): чтение таблицы → черновик + отчёт «что
  распозналось». Тесты на четырёх сгенерированных файлах: наш формат, произвольный,
  немаппируемый, смешанные типы ячеек (бинарники в репозиторий не кладём — тест сам
  собирает xlsx).
- Приём `.xlsx` в обработке входящих файлов (сейчас он отвергается на входе).
- `src/bot.ts`: кнопки развилки, автоопределение по расширению, поле «путь» в сессии,
  ветка «импорт вместо LLM» при построении черновика (прогресс-тикер не нужен —
  импорт занимает меньше секунды).
- Страховка от регресса: путь сборки не трогаем вообще (весь vitest, smoke-прогоны,
  canary); всё, что связано с Geonline (`clientId === 'geonline'`, его таблица),
  не затрагивается вовсе.

Оценка: ~1 день разработки + мини-ревью.

## 6. Четыре вопроса Тимуру (ответить до старта кода)

1. **Есть образец реального клиентского Excel?** Словарь синонимов откалиброван по
   таблице Geonline. Если под рукой стратегия AIPLUS или другого клиента в Excel —
   скинь, положим в тесты. (Не блокер: без образца тоже норм, покроем разумный дефолт.)
2. **Стратегия текстом — через обычную сборку, ок?** (см. §4.2; альтернатива —
   отдельный парсер без LLM, я против: хрупко и дублирует существующее).
3. **Автоопределение по расширению + необязательные кнопки — ок?** (альтернатива —
   только явные кнопки, но это лишний клик Азизе на каждом онбординге).
4. **Гипотезы при импорте не генерируем — ок?** (альтернатива — после импорта
   предлагать кнопку «🧠 Досинтезировать гипотезы из этого же файла» через LLM;
   это +1 небольшая задача).

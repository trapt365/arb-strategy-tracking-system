## 1. Суть проблемы (Issue Summary)

Epic 7 (F0-онбординг + мультиклиентность) завершён и live-verified: черновик AIPLUS собран (168 с, вход ~168k токенов), таблица создана (OKR 27 · гипотезы 24 · участники 15), реестр клиентов работает. Однако прод-демо показало, что **приёмка Ф2 в текущем виде невозможна**:

1. **P0-блокер:** созданная таблица клиента показывает данные Geonline. `F0_SHEETS_TEMPLATE_ID` указывает на копию боевой таблицы Geonline («Geonline Стратегический трекинг v2.0 (14)»), а не на чистый бланк. F0 (`src/f0-sheets.ts`) пишет данные клиента только в скрытые машинные листы `_okr`/`_stakeholder_map`/`_hypotheses`; десятки видимых человекочитаемых листов (`Vision&Strategy`, персональные `OKR Дамир/Шынгыс/…`, `Банк гипотез идей задач`) копируются из шаблона как есть → каждый новый клиент получает дашборды Geonline. Пилот с реальным клиентом в таком виде невозможен (и это утечка данных Geonline третьим лицам).
2. **UX-долги, блокирующие самостоятельность Азизы:** бренд бота = «Geonline» (имя одного из клиентов), простыни таблиц в Telegram нечитаемы, вход в бот — статический welcome без навигации по клиентам, онбординг не принимает Excel и не различает «импорт готовой стратегии» vs «синтез из сырых документов».

Дополнительно — гигиена: в deferred-work.md запись о падении F0-черновика продублирована (открытая копия при уже RESOLVED-баге).

## 3. Рекомендуемый подход

Новый **Epic 8 «Прод-фидбэк онбординга»**, 5 stories в порядке строгого приоритета (8.1 → 8.5), лёгкий формат solo MVP: 3 AC, ~5 задач, 1 проход ревью. Логика порядка:

- **8.1 (P0)** — без чистого шаблона пилот невозможен вообще; блокирует приёмку и создаёт риск утечки данных Geonline.
- **8.2** — бренд «Geonline» на боте недопустим при онбординге второго клиента (клиент видит чужое имя).
- **8.3** — читаемость доставки: Азиза не примет черновик, который нельзя прочитать.
- **8.4** — навигация по 2+ клиентам: без меню/селектора самостоятельная работа с двумя клиентами неудобна, но возможна.
- **8.5** — импорт готовой стратегии: ускоряет онбординг, но текущий LLM-путь уже работает (AIPLUS прошёл). Требует дизайн-развилки ДО кода.

8.1–8.3 — необходимое условие приёмки Ф2; 8.4–8.5 — качество самостоятельной работы Азизы.

## 6. UX-walkthrough онбординга в роли трекера (v2, 2026-07-08)

Пройден полный путь Азизы по коду (src/bot.ts, f0-onboarding.ts, f0-fill.ts, f0-sheets.ts, f0-client-card.ts): `/start` → `/newclient` → файлы → `/draft` → черновик → дозаполнение → `/confirm` → таблица → `/status` → `/report`.

### Что удобно (сохранить как есть)

- Подтверждение каждого файла со счётчиком пакета + inline-кнопка «✅ Собрать черновик» — приём пакета понятен без чтения инструкций.
- Ошибки сборки не разрушают пакет («Пакет цел — поправь файлы и собери снова: /draft») — повтор дёшев.
- Прогресс `(i/N)` в вопросах дозаполнения + подсказка `/skip`/`/confirm` в каждом вопросе.
- `/confirm` идемпотентен: повторный запуск не плодит дубли таблиц (existingSpreadsheetId + batchClear); реестр переиспользует таблицу той же компании.
- Персист сессии после каждого ответа — рестарт бота не теряет прогресс дозаполнения.

### Вывод walkthrough

Исходный Epic 8 покрывал прод-фидбэк (W1, W4, W8, W11 + P0-шаблон), но пропускал операционный риск **W2+W3**: на втором пилоте Азиза с высокой вероятностью примет 10-минутную сборку за зависание и убьёт собственный прогресс повторным `/newclient` — это прямой удар по критерию «Азиза сама заводит 2 клиентов».

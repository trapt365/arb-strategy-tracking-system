## 1. Единая точка входа: «выбери способ» после профиля клиента

Онбординг начинается с обязательного шага **«Профиль клиента»** (story 9.1, CR-3) — двухуровневого (решение Тимура): критический минимум 🔑 (название, суть бизнеса, топы: должность/полномочия/зона, decision maker — то, без чего таблица не собирается) обязателен; расширенный опросник (история/партнёры, финансовая Точка А, оргструктура файлом, запрос и хотелки) — опционально после минимума. Вопросы — `docs/onboarding-questionnaire-v1.0.md`, Часть A. Только после 🔑-минимума — ОДИН экран выбора способа:

```
Профиль клиента (9.1) — обязательный первый шаг
        │  профиль = grounding: имена, должности, название компании
        ▼
«Как заводим стратегию?»
 ├─ 📥 Готовая стратегия в Excel — прямой импорт (без ИИ-пересборки)
 ├─ 💬 Вопросник — соберём стратегию из ответов (можно голосом)
 └─ 📄 Документы — протоколы, транскрипты, презентации (PPTX/PDF), соберу сам
        ▼  все три пути производят ОДИН артефакт
черновик F0FullExtraction → саммари → 🔴-вопросы дозаполнения → /confirm → Google Sheets
```

Автодетект по первому файлу сохраняется (8.5): `.xlsx` → импорт, `.md/.docx/.pdf/.pptx` → документы; кнопки не обязательны. Смешивание путей в одной сессии — по-прежнему нельзя (переключение с подтверждением).

## 2. Сходимость: три пути → один артефакт

Ничего после черновика не дублируется: саммари, 🔴-проверки, диалог дозаполнения (7.3 + 8.6), `/confirm`, выгрузка в Sheets — общие для всех путей (механика 8.5 уже так устроена, `deliverF0Draft`). Результат любого пути один: рабочий инструмент трекера — таблица + контекст для недельного цикла.

**Grounding из профиля (CR-3 → CR-4в):** профиль клиента — единственный источник имён и названий для ВСЕХ путей. LLM-синтез получает список участников в промпт; транскрипт без разметки говорящих атрибутируется по этому списку; в артефактах — правильные имена из профиля, не ASR-искажения и не выдумки. Персональные листы Sheets — по топам профиля.

## 3. Пути (а) и (в): импорт Excel + презентации в «документах»

- **Импорт (а) = только Excel** — как в 8.5 (код, без LLM); словарь синонимов пополняется по реальному файлу ARB Solutions (он же идёт в тесты) — фикс сегодняшнего `import_unmappable`.
- **Презентации — в путь «документы» (в), решение Тимура §7.3:** путь называется честно, без «маски импорта». PPTX/PDF-презентация принимается как ещё один формат документа: текст извлекаем кодом (PDF — `pdf-parse`, уже в deps; PPTX — распаковка zip + текст из XML, `jszip` — одна маленькая новая зависимость, ~50 строк), дальше — существующий LLM-синтез с grounding профиля. Если пакет состоит из одной презентации готовой стратегии, промпт ужесточается: «это готовая согласованная стратегия — перенеси, не досочиняй».

## 4. Путь (б): вопросник

Профиль уже собран (9.1) → вопросник спрашивает ТОЛЬКО про стратегию, без повторов:

1. Направления (цели года) — по одному, «готово» кнопкой.
2. По каждому направлению: KR в формате «с X до Y к сроку» (валидация чисел из 8.6 переиспользуется).
3. Ответственный за KR — **кнопками из списка топов профиля** (не свободный текст — нет ASR/опечаток).
4. Гипотезы — принцип «вопрос → верифицируемая гипотеза»; список и формулировки вопросов — по **вопроснику онбординга `docs/onboarding-questionnaire-v1.0.md`** (создан 2026-07-09 из материалов Vault практики: методичка Verum к установочной встрече + опросники установочных/стратегических сессий; решение Тимура §7.1). Вопросник покрывает и шаг «Профиль клиента» (9.1) — описание клиента, текущее состояние, ожидания, проблематизация.

Ответы наполняют тот же черновик; недосказанное штатно уходит в 🔴 и дозаполнение. Механика 7.3 (persist, `/skip`, `/resume`) переиспользуется — вопросник переживает рестарт.

## 5. Голосовые ответы (CR-6) — в вопроснике, в профиле клиента и в диалоге дозаполнения

- **Пайплайн:** voice-сообщение Telegram → `getFile` (лимит TG 20 МБ) → скачать `.oga` → **существующий Soniox-адаптер** (`src/adapters/soniox.ts`: uploadFile → poll → transcript; ogg/opus поддержан) → текст → тот же обработчик, что печатный ответ. Новая транскрипция НЕ строится.
- **Подтверждение:** голос → черновик ответа «Понял так: „…“ [✅ Ок · ✏️ Править · 🎤 Заново]» — в таблицу попадает только подтверждённый текст. Ошибка распознавания не уезжает в артефакты молча.
- **Лимиты:** до 5 мин на один ответ (дольше — просьба разбить); распознавание ru+kk — Soniox валидирован на code-switching ещё в story 0.1. Стоимость ~$0.002/мин, задержка секунды.
- **Где принимается голос (решение Тимура §7.2):** вопросник, диалог дозаполнения И профиль клиента (9.1) — оргструктуру проще наговорить. В подсказках этих диалогов трекеру явно говорится: «можно ответить голосом 🎤 или прислать документ 📎» — голос и файлы не спрятаны.
- Диаризация голосовых ответов не нужна (один говорящий — трекер).

## 6. Что сознательно НЕ делаем

1. Не принимаем голос в F1-трекинг-цикле — там уже есть `/report` с записью встречи.
2. Не переносим оформление презентаций — только контент.
3. Не строим кодовый парсер структуры PPTX-слайдов (маппинг «слайд → KR») — LLM-синтез это уже умеет.
4. Не смешиваем пути в одной сессии (правило 8.5 сохраняется).

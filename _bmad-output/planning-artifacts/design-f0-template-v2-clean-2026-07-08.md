# Дизайн: чистый шаблон «Стратегический трекинг v2.0» и механика создания таблицы клиента

> Вход для Story 8.1 (Epic 8). Основан на разборе реальной выгрузки `Стратегический трекинг v2.0 — AI Plus (AI+) (2026-07-08).xlsx` (orphan-таблица AIPLUS = копия боевой Geonline + машинные листы с данными AIPLUS). Цель дизайна — облегчить еженедельную работу трекера с командой клиента.

## 0. Что показал разбор выгрузки

- **27 листов**, из них 13 скрытых legacy-листов Geonline (`Vision&Strategy`, `ОКР на 2-ое полугодие`, `OKR Дамир/Шынгыс/Максат/…`, `Банк гипотез идей задач`, `Ограничения узелки`) и 10 видимых листов — **тоже Geonline** («📊 ВСЕ OKR» с заголовком «🎯 GEONLINE: ВСЕ OKR…», персональные листы по фамилиям Geonline-топов). Данные AIPLUS — только в 4 машинных листах `_okr`(27 KR)/`_stakeholder_map`(15)/`_hypotheses`(24)/`_f5_metrics`(пусто).
- **Рабочая единица трекера — персональный лист топа** из 4 зон: (1) шапка (должность, BSC, ответственность, период, «Последняя встреча» + итоги, «Следующая встреча» + повестка); (2) 🎯 Панель OKR топа (№ KR, краткое название, формулировка, текущий статус, целевая метрика, прогресс, срок, комментарий); (3) ⚡ Активный трекинг инициатив (тип 📌задача/📦проект/🔬гипотеза/💡идея, связь с KR, ответственный, срок, Неделя 1–4, комментарии) + 💡 бэклог; (4) 📚 легенда методологии (типы, статусы 🟢🟡🔴✅🔄⏸️⚠️💭🔮, правила ритма, цвета BSC).
- **Схемы машинных листов беднее рабочих**: в `_hypotheses` нет ответственного/статуса/срока/связи с OKR (в legacy «Банке гипотез» они были); в `_stakeholder_map` нет telegram-контакта (он собирается диалогом, но уезжает только в card.json).

## 1. Решение: Apps Script или копия шаблона? → Копия + Sheets API, шаблон-как-код

**Apps Script для СОЗДАНИЯ файла не нужен.** Причины:

1. У бота уже есть OAuth-клиенты Drive/Sheets write (story 7.4). Единственная динамическая задача, которую не решают формулы, — размножение персональных листов по топам — закрывается штатным `batchUpdate({duplicateSheet})`.
2. Вызов Apps Script из бота (Execution API) требует deployment, привязки к GCP-проекту и **интерактивной авторизации каждым пользователем с диалогом «непроверенное приложение»** — прямое трение для Азизы, против якоря эпика.
3. Формулы дают live-пересчёт панелей без всякого скрипта: трекер правит данные — панели обновляются сами.

**Но идея «создавать файл скриптом» верна на уровне ШАБЛОНА:** вместо ручного вылизывания бланка — репозиторный скрипт `scripts/f0-build-template.ts` (Sheets API, те же OAuth-креды), который строит/обновляет эталонный шаблон с нуля: листы, заголовки, формулы, легенду, форматирование, протекции. Шаблон перестаёт быть хрупким ручным артефактом и версионируется кодом (изменение схемы = правка скрипта + перегенерация). Это и есть ответ на «gs-скрипт»: генератор — да, но наш TypeScript-скрипт в репо, не Apps Script внутри таблицы.

**Apps Script остаётся Growth-опцией** для интерактива внутри таблицы (меню трекера: «отметить встречу», «добавить топа», «пересобрать панели») — container-bound скрипт копируется вместе с шаблоном автоматически. Не MVP.

## 2. Принцип дизайна: машинные листы — единственный источник, панели — формулы

```
трекер правит ДАННЫЕ (⚙️)  ──►  панели (📊/👤) пересчитываются сами  ──►  команда и CEO смотрят панели
                └──►  F1-пайплайн (readClientContext) читает те же ⚙️-листы — без двойного ввода
```

Это решает сразу три проблемы: прод-баг 8.1 (панели рендерятся под клиента автоматически), двойной ввод (сейчас трекер правил бы и панель, и `_okr`), протухание контекста F1 (пайплайн видит те же статусы, что и команда).

Следствие: машинные листы **не прячем**, а оформляем как рабочие «⚙️-данные» (тёмная шапка, закреплённая строка, фильтры) — трекер правит их напрямую. Панели защищаем warning-протекцией (правка с предупреждением).

## 3. Состав листов чистого шаблона

### Видимые панели (формульные, protected-warning)

| Лист | Содержимое | Источник |
|---|---|---|
| «📊 Все OKR» | Шапка: компания, период, счётчики (OKR/KR/участники). Таблица «Участники и нагрузка» (имя, должность, BSC, зоны, число KR). Детальный список KR: одна сортированная таблица с колонкой «OKR-группа» + закреплённые фильтры (отказ от раздельных блоков-простыней ради простых формул) | `⚙️ _meta`, `_stakeholder_map`, `_okr` |
| «👤 {Имя топа}» ×N | 4 зоны как в текущем рабочем листе: шапка топа (роль/BSC — VLOOKUP из `_stakeholder_map`; последняя/следующая встреча — из «📅 Лог встреч»); 🎯 панель KR топа — `FILTER(_okr; owner=$B$1)`; ⚡ инициативы топа — `FILTER(⚙️ _hypotheses; owner=$B$1)`; 📚 легенда (статика) | `_okr`, `_hypotheses`, лог встреч |
| «🧪 Банк гипотез» | Все гипотезы: формулировка, ЕСЛИ-ТО-ПОТОМУ ЧТО, метрика, связь с OKR, ответственный, срок, статус, пометка ⚠️synthesized | `_hypotheses` |
| «📅 Лог встреч» | Рабочий (ручной append трекером; позже — ботом из F1): дата, топ, итоги встречи, договорённости, повестка следующей | ручной ввод |
| «📋 Ограничения (узелки)» | Пустая структура по методологии TOC (колонки из legacy: приоритет, ограничение, тип TOC, score, влияние, контекст, топ-решение) — ведётся трекером руками | ручной ввод |
| «📖 Методология» | Легенда целиком (типы инициатив, статусы, правила ритма, цвета BSC) — статика, единственный источник легенды (из персональных листов можно ссылаться) | статика |

### Машинные листы (⚙️-данные, редактируемые трекером)

- `_meta` (новый): key/value — company, period, onboarding_date, tracker. Панели берут шапку отсюда; F0 заполняет при онбординге.
- `_okr` — схема как есть (`kr_number, short_name, key_result, owner, owner_position, current_status, target, progress, deadline, okr_group, quarter`); `current_status` = база «с X» на момент онбординга, дальше — живой статус.
- `_stakeholder_map` — **+ колонка `telegram`** (сейчас контакт из диалога дозаполнения уезжает только в card.json).
- `_hypotheses` — **+ колонки `okr_link, owner, deadline, status`** (F0 пишет null там, где не извлёк; трекер дозаполняет; иначе «Банк гипотез» неполноценен как рабочий инструмент).
- `_f5_metrics`, `_ops_logs` — без изменений.
- «👤 Шаблон топа» — скрытый эталонный лист с формулами, параметризованными по `$B$1` (имя топа).

Запись header-driven (как в 7.4) — добавление колонок в шаблон безопасно, код пишет по именам заголовков.

## 4. Механика создания таблицы клиента (изменения `src/f0-sheets.ts`)

1. `files.copy(F0_SHEETS_TEMPLATE_ID)` — как сейчас.
2. Запись `_meta` + `_okr` + `_stakeholder_map` + `_hypotheses` — как сейчас (header-driven, batchClear перед записью).
3. **Новое:** для каждого уникального `owner` из `_okr` (∪ топы из `_stakeholder_map` с KR): `duplicateSheet(«👤 Шаблон топа»)` → rename «👤 {Имя}» → `$B$1` = имя → показать лист. Идемпотентность: лист с таким именем уже есть → пропустить (retry `/confirm` без дублей сохраняется).
4. Скрыть эталонный лист; порядок листов: панели → лог → данные.

Формулы — RU-локаль (разделитель `;`), эскизы:

```
KR топа:            =IFERROR(FILTER('_okr'!A2:K; '_okr'!D2:D="{$B$1}"); "Нет KR")
Число KR участника: =COUNTIF('_okr'!D:D; A9)
Последняя встреча:  =IFERROR(INDEX(SORT(FILTER('📅 Лог встреч'!A2:E; '📅 Лог встреч'!B2:B=$B$1); 1; ЛОЖЬ); 1); "—")
Детальный список:   =QUERY('_okr'!A2:K; "select J,A,C,F,G,I,D where A is not null order by J,A"; 0)
```

## 5. Еженедельный ритм трекера (что дизайн облегчает)

1. **Перед встречей с топом:** открыть «👤 {Топ}» — KR со статусами, активные инициативы, итоги прошлой встречи и повестка — всё на одном экране (сейчас это же собирается руками).
2. **После встречи:** `/report` боту (F1-отчёт) + 2 минуты правок в ⚙️-данных (статусы KR, инициативы) + строка в «📅 Лог встреч» — все панели и контекст F1 обновились сами.
3. **Для команды/CEO:** «📊 Все OKR» и «🧪 Банк гипотез» всегда актуальны, шарится один URL.

## 6. Риски и решения

> **Утверждено Тимуром 2026-07-08 (correct-course):** все три открытых вопроса ниже закрыты в пользу рекомендаций.

- ✅ **РЕШЕНО: трекер правит статусы прямо в видимых ⚙️-данных** (`_okr`/`_hypotheses`), панели — read-only формулы с warning-протекцией. Альтернатива (правки в панелях) отклонена — возвращала двойной ввод.
- ✅ **РЕШЕНО: «📊 Все OKR» — единая сортированная таблица** с колонкой OKR-группы и закреплёнными фильтрами (без блоков с заголовками-строками). Фидбэк Азизы собрать по факту первого пилота, не блокирует 8.1.
- ✅ **РЕШЕНО: повторный `/confirm` поверх «пожившей» таблицы затирает правки трекера в `_okr` (batchClear)** — принято как известное ограничение MVP; фиксируется в story 8.1. Опция на будущее: предупреждение при confirm, если таблица уже редактировалась после онбординга.
- **Хрупкость FILTER при переименовании owner**: имя топа — ключ связи `_okr` ↔ персональный лист. Переименование в `_stakeholder_map` без правки `_okr.owner` рассыпает FILTER. MVP: правило в «📖 Методология»; Growth: id-ключи.
- Legacy-листы Geonline в чистый шаблон **не переносятся вовсе** (ни скрытыми): «Ограничения (узелки)» и «Методология» переносятся пустыми структурами.

# Недочёты живого прогона WP-39 (13 июля)

> **ТЗ для кодировочной сессии.** Всего 12 находок. **Блокеры ядра (приоритет):** P0 краш на файле >20 МБ (нет `bot.catch`) · P0 хрупкий парсер отчёта (пустой owner роняет весь отчёт) · P1 клиентские таблицы не расшарены на сервис-аккаунт (на прогоне обошли вручную). ## P0 — аудио >20 МБ роняет весь бот (критично)

- **Симптом:** пилот загрузил `жанель 11.m4a` (42 мин, ~27 МБ). Telegram `getFile` → «400: file is too big» (лимит Bot API = 20 МБ). Ошибка не поймана → **процесс упал** («Telegram bot failed to start / Stopping bot»), pm2 перезапустил.
- **Два корня:**
  1. **Нет глобального обработчика ошибок** (`bot.catch` не установлен — в логах «No error handler was set!»). Любая ошибка в хендлере убивает прод.
  2. Файлы >20 МБ не обрабатываются graceful — нет проверки размера до `getFile`, нет деградации.
- **Где:** `handleMeetingFileIntake` — [bot.js:3535](~/apps/strategy-tracking-system/dist/bot.js#L3535).
- **Фикс:** (а) поставить `bot.catch`; (б) проверять `file_size` до `getFile`, при >20 МБ — сообщить клиенту («файл великоват, сожми/разбей») или качать через local Bot API server / клиентский MTProto.

## P1 — F5 трекер гипотез: 403 на чтение клиентской таблицы

- **Симптом:** `readHypothesesSheet` на таблице qubiq (`_hypotheses!A1:Z`) → 403 `forbidden_or_revoked` «The caller does not have permission». Пайплайн F5 лежит с 10.07 → watchdog шлёт «Pipeline down» + **эскалация Айдару 11.07** (проверить, реально ли ушло ему в чат).
- **Корень:** таблицы клиента создаются под user-OAuth (обход квоты сервис-аккаунта), а F5 **читает** под сервис-аккаунтом `geonline-tracking-bot@geonline-report-bot...` — доступ ему не выдан. Разъезд read/write identity.
- **Где:** [sheets.js:450 readHypothesesSheet](~/apps/strategy-tracking-system/dist/adapters/sheets.js#L450), `runHypoTracker` f5-hypo-tracker.js:331.
- **Фикс:** при создании клиентской таблицы шарить её на email сервис-аккаунта (editor), либо все read-операции вести под тем же OAuth, что и write.
- **Повтор на прогоне 13.07:** тот же 403 заблокировал **встречу→отчёт** (`[F1/sheets.read]` листы `_okr`/`_stakeholder_map`/`_f5_metrics` таблицы geonline) и недельный отчёт — бот ушёл в «Задержка, пиши вручную». Подтверждает: **все** read-пайплайны (F1 отчёт, F5 трекер) ломаются на новых клиентских таблицах. Приоритет фикса — высокий, это блокер ядра для любого нового клиента. Ручной обход: расшарить таблицу на `geonline-tracking-bot@geonline-report-bot.iam.gserviceaccount.com` (editor).

## P0 — отчёт по встрече падает: хрупкий парсер _okr + пустые owner

- **Симптом:** после починки прав (403→ок) отчёт упал с `sheets:invalid_value`. - **Фикс:** (а) парсер терпит пустой owner (дефолт «—» / warn, не throw); (б) онбординг проставляет owner по умолчанию или помечает как пробел для дозаполнения, не пустой. У свежего клиента пробелы — норма, отчёт должен собираться поверх них.

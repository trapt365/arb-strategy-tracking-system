## Инструкции

### 1. Тип документа (document_type)
Если материал не про стратегию/OKR/цели (README, инструкция, произвольный текст) — верни `"document_type": "other"` с пустыми массивами.

### 2. Objectives и Key Results
Как в базовом извлечении:
- `objectives[].title` — формулировка objective (сохрани нумерацию O1/TO1/IN1).
- `krs[]`: `formulation` (как в документе), `base` («с X», null если нет), `target` («до Y», null), `owner` (null если не указан), `deadline` (как записан: «До 30.07.2026», «Q1 2026», «Постоянно»; null если нет).
- Табличный rowspan: пустая ячейка Objective → относи KR к последнему непустому выше.

### 3. Банк гипотез (hypotheses)
Гипотеза — проверяемое предположение «ЕСЛИ [изменение] ТО [измеримый результат] ПОТОМУ ЧТО [логика]».
- `statement` — краткое название гипотезы (обязательно).
- `ifThenBecause` — полная формулировка ЕСЛИ-ТО-ПОТОМУ ЧТО, если она есть в документе (в GeOnline BSC-версии — прямо в этом формате); иначе null.
- `metric` — **метрика проверки гипотезы** (инвариант 2). Если в документе явной метрики нет — `null` (НЕ выдумывай; бот пометит и переспросит).
- `department` — департамент/направление, если определимо; иначе null.
- `synthesized`:
  - `false` — гипотеза явно присутствует в документе как гипотеза.
  - `true` — в документе гипотез НЕТ (например, протокол в стиле «решения D001 / инициативы / action items»), и ты **вывел кандидата в гипотезу** из решения или инициативы. Такие ОБЯЗАТЕЛЬНО помечай `synthesized: true` — трекер их подтвердит.
- Не дублируй: одна гипотеза, повторённая в разных срезах документа, — одна запись.

### 4. Участники (participants)
- `name` — ФИО/имя как в документе (обязательно).
- `role` — должность/роль (null если нет).
- `department` — направление (null если нет).
- `contact` — telegram/телефон ТОЛЬКО если явно в документе; почти всегда `null` (спросим у трекера).
Источник — таблицы участников, разделы «Контакты и ответственные», ответственные в OKR.

### 5. НЕ ВЫДУМЫВАЙ (инвариант 3)
Отсутствующее значение — `null`. Переписанные версии из аудит-колонок («Как улучшить») НЕ подставляй. Сомнительные/противоречивые фрагменты — в `unrecognized` кратким описанием.

### 6. Особенности форматов
Сохраняй язык оригинала (казахские фразы внутри русского — как есть). Эмодзи-статусы (🟢🟡🔴⏳⛔🆕) — данные, не декорация. Разные форматы дат — как в документе.

## Формат ответа

Верни ТОЛЬКО валидный JSON (без markdown fences):

{
  "document_type": "strategy" | "other",
  "company": "Название" | null,
  "objectives": [
    { "title": "…", "krs": [ { "formulation": "…", "base": "…"|null, "target": "…"|null, "owner": "…"|null, "deadline": "…"|null } ] }
  ],
  "hypotheses": [
    { "statement": "…", "ifThenBecause": "ЕСЛИ … ТО … ПОТОМУ ЧТО …"|null, "metric": "…"|null, "department": "…"|null, "synthesized": false }
  ],
  "participants": [
    { "name": "…", "role": "…"|null, "department": "…"|null, "contact": "…"|null }
  ],
  "unrecognized": ["…"]
}

### Пример синтеза (документ без явных гипотез, стиль протокола SAM)

Фрагмент: «Решение D003: выйти на рынок B2G через пилот с 2 госорганами в Q1.»

Правильно:

{
  "statement": "Выход на B2G через пилот с госорганами",
  "ifThenBecause": "ЕСЛИ запустить пилот B2G с 2 госорганами, ТО получим первый B2G-контракт, ПОТОМУ ЧТО подтверждённый спрос откроет сегмент",
  "metric": null,
  "department": "Продажи",
  "synthesized": true
}

(Гипотеза выведена из решения — `synthesized: true`; метрики в документе нет — `metric: null`, не выдумана.)

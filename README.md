# ARB AI-Tracking System

> **Telegram-бот** на базе AI, который превращает записи коучинговых встреч в готовые отчёты, повестки и аналитику — прямо в чате

## Зачем

Большинство стратегий проваливаются не потому что плохие, а потому что у команды не поставлена **практика стратегического исполнения** — способ работы, при котором команда формулирует гипотезы, проверяет их связь с OKR, тестирует в недельном ритме и принимает решения самостоятельно.

ARB ставит эту практику через еженедельный коучинговый трекинг с живыми трекерами. Но модель упирается в потолок: контроль качества завязан на руководителе лично, трекеры перегружены рутиной отчётности, клиент не видит прогресс между встречами.

**AI-система снимает это ограничение.** Трекер, освобождённый от рутины, направляет время на то, что создаёт результат: задавать правильные вопросы, вскрывать ограничения, возвращать команду к стратегическому фокусу.

## Что делает система

Единый конвейер данных: **запись встречи → транскрипция → AI-анализ → результат для каждой роли.**

| Функция | Что делает | Для кого |
|---------|-----------|----------|
| **F1 — Отчёт** | Структурированный отчёт по встрече с цитатами и обязательствами | Трекер → Топ-менеджер |
| **F4 — Повестка** | Подготовка к следующей встрече: ключевые вопросы, незакрытые обязательства, покрытие OKR | Трекер |
| **F5 — Метрики** | Сбор 2 метрик (leading + lagging) от каждого топа через Telegram | Трекер, CEO |
| **F3 — CEO-сводка** | 5 строк + 🟢🟡🔴 статус по каждому OKR | CEO |
| **F2 — Качество коучинга** | Talk ratio, распределение ролей, покрытие OKR _(Phase 2)_ | Руководитель ARB |

### Принцип работы

```
Азиза проводит встречу с топом
        │
        ▼
Запись транскрибируется (Soniox)
        │
        ▼
Азиза отправляет ссылку боту в Telegram
        │
        ▼
AI генерирует отчёт (4 шага: извлечение → анализ → форматирование → доставка)
        │
        ▼
Азиза проверяет, правит если нужно, утверждает
        │
        ▼
Бот доставляет отчёт топу
```

**Контроль трекера** — ключевое правило: система ничего не отправляет клиенту без одобрения трекера.

## Дорожная карта

### Epic 0: Подготовка и валидация
> Тесты транскрипции, онбординг клиента, стабилизация промптов, инфраструктура

### Epic 1: Ядро — первый отчёт
> Азиза отправляет ссылку → получает отчёт → утверждает → отчёт доставлен топу

### Epic 2: Полный недельный цикл
> Понедельник: метрики (F5) + повестка (F4) + CEO-сводка (F3). Вторник–четверг: отчёты (F1)

### Epic 3: Стабилизация (4 недели на пилоте)
> Итерация промптов, мониторинг принятия системы, решение Go/No-Go

### Epic 4: Качество коучинга _(Phase 2)_
> Talk ratio, классификация ролей трекера (4 шляпы), покрытие OKR, QC-дашборд

### Epic 5: Расширенная аналитика _(Phase 2)_
> Жизненный цикл обязательств, тренды метрик, расширенная CEO-сводка

### Epic 6: Масштабирование _(Growth)_
> От 1 клиента к 3–5, PostgreSQL, CI/CD, автодоставка CEO-сводок

### Таймлайн

```
Апрель нед.1    Epic 0 — подготовка, тесты
Апрель нед.2    Epic 1 — первый отчёт ✦ Day 1
Апрель нед.3    Epic 2 — полный цикл F1+F4+F5+F3
Апр.4 – Май.2   Epic 3 — стабилизация (4 недели)
Май нед.3        Go/No-Go → решение о Phase 2
Май – Июнь       Epic 4–5 — качество коучинга + аналитика
Июнь+            Epic 6 — масштабирование
```

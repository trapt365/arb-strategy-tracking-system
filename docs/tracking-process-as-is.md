# Процесс стратегического трекинга (as-is)

> Описание текущего процесса по состоянию на март 2026. Клиент: Geonline (Дамир Самарханов, CEO).

---

## 1. Роли

| # | Роль | Кто | Метод работы |
|---|------|-----|-------------|
| 1 | **Трекер** | Азиза | Проводит еженедельные 1:1 с каждым топ-менеджером. Наблюдает динамику, фиксирует сигналы. Единственная точка контакта с командой клиента |
| 2 | **Аналитик** | Тимур | Получает транскрипты и наблюдения от трекера. Готовит executive summary и динамику гипотез для CEO. Не контактирует с клиентами напрямую — вся коммуникация через Азизу |
| 3 | **Куратор** | Айдар | Коучинг CEO (еженедельно). Ревью процесса трекинга с Азизой (периодически). Стратегический надзор |
| 4 | **CEO** | Дамир | Получатель отчётов. Принимает решения по рискам и гипотезам. Коучинг-сессии с Айдаром |
| 5 | **Команда** | 6-7 топ-менеджеров | Участники трекшн-встреч. Генерируют гипотезы, отчитываются по KR |

## 3. Недельный цикл

```
Пн–Чт    Азиза проводит 1:1 (20-30 мин каждая, 6-7 встреч)
          │
          ▼
Пт        Азиза отправляет Тимуру:
          ├── ссылки на транскрипты
          └── свои наблюдения (комментарии по неделе)
          │
          ▼
Пт–Сб    Тимур готовит:
          ├── Executive Summary (пульс OKR, топ-3 риска, метрики, фокус)
          └── Динамику гипотез (статусы по департаментам)
          │
          ▼
          Азиза передаёт отчёты → Дамиру (CEO)
```

**Ежемесячно:** Командная сессия — общий отчёт за месяц.

**Параллельно:** Айдар ↔ Дамир (коучинг, еженедельно), Айдар ↔ Азиза (ревью процесса, периодически).

## 4. Формат трекшн-встречи (1:1 Азиза ↔ топ-менеджер)

Из транскриптов выделяется следующая структура:

1. **Статус по KR** — проход по каждому KR из OKR-трекера: что сделано, что изменилось
2. **Гипотезы** — какие запущены, какие результаты, новые идеи
3. **Блокеры и запросы** — что нужно от CEO / других отделов
4. **Наблюдения трекера** — Азиза фиксирует энергию, настрой, системные паттерны

> Формат не формализован — Азиза ведёт встречу по своему сценарию, адаптируя под каждого участника.

---

## 5. Формат Executive Summary

Структура текущего отчёта (неделя 6):

1. **Общий пульс стратегии** — таблица OKR × статус × динамика
2. **Топ-3 риска** — с прямой речью участников, вопросами к CEO
3. **Диагностика трекера** — позитивные сигналы + системные ограничения
4. **Прогресс по ключевым метрикам** — факт vs цель
5. **Что движется хорошо** — буллеты
6. **Фокус на следующей неделе** — приоритеты с ответственными и сроками
7. **Новые гипотезы от команды** — таблица
8. **Вопросы к CEO** — пронумерованный список

---

## 6. Формат динамики гипотез

Группировка по департаментам. На каждую гипотезу:

| Поле | Описание |
|------|----------|
| # | Код (П-1, М-1, С-1...) |
| Гипотеза | Формулировка «если X, то Y» |
| Статус | Работает / В тесте / Не работает / Запланирована / Остановлена |
| Запуск | Когда стартовала |
| Результат / Метрика | Факт |
| Следующий шаг | Конкретное действие |

Сводная таблица в конце: статусы по департаментам.

## 8. PRD: целевое состояние (to-be vision)

В марте 2026 подготовлен PRD (`_bmad-output/planning-artifacts/prd.md`) — описание AI-системы для автоматизации и масштабирования трекинга.

**Суть:** единый data pipeline (транскрипт → AI-анализ → метрики) с тремя output-форматами:

| Потребитель | Что получает | Ценность |
|-------------|-------------|----------|
| **Трекер (Азиза)** | Автоотчёт по встрече + повестка на неделю + фидбэк по методологии | Экономия 2ч/встречу, accountability, рост мастерства |
| **Руководитель (Айдар)** | QC-скоринг: talk ratio, покрытие OKR, распределение шляп | Контроль качества всех трекеров на одном экране |
| **CEO (Дамир)** | Сводное уведомление: 5 строк + статус OKR + 1 инсайт | Прозрачность прогресса между встречами |

**Фазы MVP:**
- **Phase 1** (нед. 1-4): F1 автоотчёт + F4 повестка + F3-lite уведомление CEO
- **Phase 2** (нед. 3-6): F2 QC-скоринг + F3-full агрегация + фидбэк трекеру

**Ключевая метрика:** не эффективность pipeline, а **скорость обучения команды** — тренд улучшения качества гипотез.

**Стек MVP:** tldv + Google Sheets + Claude API + Telegram Bot.

## 9. FPF-ревью PRD (2026-03-25)

PRD прогнали через FPF-проверку. Выявлены 4 дыры:

### 9.1. «Оргвозможность» → «Практика стратегического исполнения» (решено)

PRD использовал «оргвозможность» в трёх смыслах. Решение — замена на точные термины из СМ-онтологии:

> **Практика стратегического исполнения** — способ работы команды: формулировать гипотезы, проверять их связь с OKR, тестировать в недельном ритме, принимать решения без зависимости от CEO.

| Смысл | Старый термин | Новый термин |
|-------|--------------|-------------|
| A (характеристика) | «оргвозможность» | **Практика стратегического исполнения** — способ работы команды |
| B (результат трекера) | «создаёт оргвозможность» | **Постановка практики** — команда освоила практику |
| C (метрика) | «скорость роста оргвозможности» | **Уровень владения практикой** — от «не владеет» до «автономно исполняет» |

### 9.2. Роли переименованы по методу работы (решено)

| Было (по должности) | Стало (по методу работы) | Суть |
|---------------------|-------------------------|------|
| Трекер (Азиза) | **Коуч практики** | Формирует навык через вопросы, наблюдает, фиксирует |
| Аналитик (Тимур) | **Аналитик практики** | Структурирует данные, готовит отчёты, строит pipeline |
| Руководитель (Айдар) | **Контроллер качества коучинга** | QC трекеров, коучинг CEO, стратегический надзор |
| CEO (Дамир) | **Заказчик постановки практики** | Получатель результата, принимает решения |
| Команда | **Команда-в-обучении** | Осваивают практику стратегического исполнения |

### 9.3. Склейки характеристика/предпочтение в бизнес-метриках

- «Азиза расстроилась бы если выключили» — эмоция, не метрика
- «Доверие к качеству» — предпочтение по характеристике «оценка quality»
- «Трекер приходит подготовленным» — предпочтение, не характеристика

### 9.4. Жизненный цикл гипотезы — формализован (решено)

#### Критерии правильной гипотезы (3 критерия)

| # | Критерий | Тест | Пример (хороший) | Антипример |
|---|----------|------|-------------------|------------|
| 1 | **Связь с KR** | Гипотеза явно привязана к конкретному KR | «Видеозвонки повысят конверсию (KR-2.3)» | «Попробуем больше звонить» |
| 2 | **Измеримость** | Есть метрика и порог успеха | «Конверсия с 25% до 60%» | «Станет лучше» |
| 3 | **Срок проверки** | Определён период тестирования | «2 недели» | Без срока |

#### Жизненный цикл (статусы + критерии переходов)

```
Идея → [проверка 3 критериев] → Запланирована → [запуск] → В тесте → [срок истёк]
                                                                         ↓
                                                              ┌──────────┼──────────┐
                                                              ↓          ↓          ↓
                                                          Работает   Не работает  Неопределённо
                                                              ↓          ↓          ↓
                                                        Масштабировать  Закрыть   Продлить/Pivot
```

| Переход | Критерий перехода | Кто решает |
|---------|-------------------|------------|
| Идея → Запланирована | Проходит все 3 критерия (KR, метрика, срок) | Топ-менеджер + коуч практики |
| Запланирована → В тесте | Первое действие выполнено | Топ-менеджер |
| В тесте → Работает | Метрика достигла порога к сроку | Аналитик практики (по данным) |
| В тесте → Не работает | Метрика не достигнута к сроку | Аналитик практики (по данным) |
| В тесте → Неопределённо | Срок истёк, данных недостаточно | Коуч + топ-менеджер решают: продлить или pivot |
| Любой → Остановлена | Внешний блокер или смена приоритетов | CEO / топ-менеджер |

> **Ключевое отличие от текущего:** сейчас «В тесте» — бессрочный статус. С формализацией — срок проверки обязателен, по истечении принимается решение.

## 10. Узкие места и потери

### 10.1. Сбор данных (Пн–Чт: коуч практики проводит 1:1)

| Узкое место | Потеря | Масштаб |
|---|---|---|
| **Формат встречи не формализован** | Адаптация под каждого — хорошо для коучинга, плохо для воспроизводимости. Новый трекер не сможет повторить | Блокирует масштабирование |
| **Нет повестки** | Подготовка по памяти + Excel (~30 мин/встречу). Слепые зоны: KR не обсуждавшиеся 2+ недели пропускаются | ~2.5h/нед на подготовку |
| **Обязательства не фиксируются** | Топ-менеджер сказал «сделаю к пятнице» — нигде не записано до отчёта аналитика. Нет accountability между встречами | Потеря follow-up |

### 10.2. Передача данных (Пт: коуч → аналитик)

| Узкое место | Потеря | Масштаб |
|---|---|---|
| **Мессенджер без структуры** | Ссылки на транскрипты + наблюдения приходят в произвольном формате. Аналитик собирает по кускам | 30–60 мин/нед на сборку |
| **Транскрипты сырые** | Code-switching (каз/рус), ошибки распознавания, нет разметки по спикерам. Аналитик фильтрует шум | ~1h/нед на парсинг |
| **Наблюдения — свободная форма** | Нет шаблона для коуча. Качество зависит от загрузки на неделе | Нестабильное качество входа |

### 10.3. Сборка отчётов (Пт–Сб: аналитик)

| Узкое место | Потеря | Масштаб |
|---|---|---|
| **Ручная сборка промпта** | Аналитик читает 6-7 транскриптов, пишет промпт, проверяет output LLM, правит | **~4h/нед на клиента — главное узкое место масштабирования** |
| **Динамика гипотез обновляется вручную** | 54 гипотезы × статусы × метрики. Аналитик вручную извлекает из транскриптов | ~1h/нед, растёт линейно |
| **Нет шаблона проверки output** | Аналитик проверяет LLM-отчёт по интуиции, нет чеклиста качества | Риск пропуска ошибок |

### 10.4. Доставка (коуч → заказчик)

| Узкое место | Потеря | Масштаб |
|---|---|---|
| **Нет обратной связи от CEO** | Отчёт отправлен — прочитал? Отреагировал? Неизвестно | Вопросы к CEO копятся без ответов |
| **Нет адаптации формата** | Отчёт ~200 строк. CEO в кризисе — возможно, не читает дальше 1-го экрана | Потеря ценности нижних секций |

### 10.5. Системные (весь цикл)

| Узкое место | Потеря | Масштаб |
|---|---|---|
| **SPOF: коуч (Азиза)** | Единственный трекер. Болезнь/отпуск = остановка трекинга | Блокирует масштабирование |
| **SPOF: аналитик (Тимур)** | Единственный аналитик. ~4h/нед на клиента = потолок 2-3 клиента | **Главный блокер бизнес-модели** |
| **QC контроллера — неформальный** | Периодические созвоны Айдара с Азизой, нет метрик качества коучинга | Нет раннего обнаружения проблем |
| **OKR-трекер в Excel** | Ручное обновление, нет интеграции, нет истории изменений | Рутина + риск рассинхрона |
| **Нет институциональной памяти** | Контекст клиента — в головах Азизы и Тимура. Смена трекера = потеря контекста | Блокирует масштабирование |

### Топ-3 по влиянию

1. **Аналитик = бутылочное горлышко** (~4h/клиент/нед) — потолок 2-3 клиента
2. **Нет повестки для коуча** — слепые зоны, потеря follow-up
3. **Нет обратной связи от CEO** — разрыв цикла

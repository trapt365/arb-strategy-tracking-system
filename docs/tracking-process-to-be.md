## 3. Недельный цикл (to-be)

```
Пн        AI отправляет коучу повестку на неделю (F4)
утро      ├── 3 пункта на каждого топа
          ├── незакрытые обязательства
          └── слепые зоны (KR без обсуждения 2+ нед)
          │
          ▼
Пн–Чт    Коуч проводит 1:1 (20-30 мин)
          После каждой встречи:
          ├── tldv автоматически транскрибирует
          ├── Коуч отправляет боту ссылку (1 шаг)
          ├── AI генерирует автоотчёт (F1) за ~10 мин
          └── Коуч проверяет → отправляет топ-менеджеру
          │
          ▼
Пт        AI собирает агрегацию за неделю:
          ├── Executive Summary (F3-full)
          ├── Динамика гипотез (автообновление статусов)
          ├── QC-скоринг (F2) → контроллеру
          └── Фидбэк коучу по методологии
          │
          ▼
          Коуч проверяет → отправляет:
          ├── F3-full → заказчику (CEO)
          └── F3-lite (5 строк) → заказчику в Telegram (Пн утром)
```

**vs as-is:**

| Этап | as-is | to-be | Экономия |
|------|-------|-------|----------|
| Подготовка к встрече | 30 мин/встречу (память + Excel) | 2 мин (чтение повестки F4) | ~2.5h/нед |
| Обработка встречи | не делалось до конца недели | 10-15 мин после встречи (проверка F1) | Реальное время vs задержка 3-5 дней |
| Сборка отчёта | ~4h/нед (Тимур вручную) | ~30 мин/нед (ревью AI-output) | ~3.5h/нед |
| Обновление гипотез | ~1h/нед (вручную) | Автоматически | ~1h/нед |
| QC коучинга | Неформально | Автоматический скоринг | Из «не было» → есть |

## 4. Покрытие узких мест (as-is §10 → to-be)

### 10.1. Сбор данных

| Узкое место | Решение (to-be) | Функция |
|---|---|---|
| Формат встречи не формализован | Повестка с направляющими вопросами задаёт структуру | F4 |
| Нет повестки | AI генерирует повестку на основе предыдущих встреч + OKR | F4 |
| Обязательства не фиксируются | Автоотчёт с цитатами = accountability | F1 |

### 10.2. Передача данных

| Узкое место | Решение (to-be) | Функция |
|---|---|---|
| Мессенджер без структуры | Коуч отправляет боту ссылку на tldv — один шаг | Telegram Bot |
| Транскрипты сырые | tldv API + разметка спикеров | Pipeline |
| Наблюдения — свободная форма | Шаблон наблюдений для коуча | Процесс |

### 10.3. Сборка отчётов

| Узкое место | Решение (to-be) | Функция |
|---|---|---|
| Ручная сборка промпта | AI pipeline: транскрипт → structured output автоматически | F1 + F3 |
| Динамика гипотез вручную | AI извлекает из транскриптов → обновляет трекер | Pipeline |
| Нет шаблона проверки output | Structured output с цитатами = встроенная верифицируемость | F1 |

### 10.4. Доставка

| Узкое место | Решение (to-be) | Функция |
|---|---|---|
| Нет обратной связи от CEO | F3-lite: 5 строк + 1 вопрос. Трекинг: открыл/не открыл | F3-lite |
| Нет адаптации формата | Адаптивный формат (сокращать при низкой вовлечённости) | F3-lite |

### 10.5. Системные

| Узкое место | Решение (to-be) | Функция |
|---|---|---|
| SPOF: коуч | F4 + шаблоны + институциональная память = онбординг нового трекера | F4 + F0 |
| SPOF: аналитик | Автоматизация pipeline: с ~4h → ~1-2h/клиент/нед | F1 + F3 |
| QC неформальный | QC-скоринг: talk ratio, покрытие OKR, шляпы | F2 |
| Excel без интеграции | Google Sheets + авто-обновление из pipeline | Pipeline |
| Нет институциональной памяти | Стейкхолдерная карта (F0) + история сессий в pipeline | F0 |

## 5. Непокрытые риски (to-be создаёт новые)

| # | Риск | Описание | Митигация |
|---|------|----------|-----------|
| 1 | **AI галлюцинации** | LLM может приписать обязательства, которых не было | Structured output + цитаты + контроль коуча |
| 2 | **Зависимость от tldv** | Single provider для транскрибации | Архитектурно: вход = текст, не tldv API. Fallback: ручной экспорт |
| 3 | **Качество транскрибации** | Code-switching каз/рус, шум | Контрольная точка Day 1 |
| 4 | **Коуч → оператор pipeline** | Азиза тратит время на правку AI вместо коучинга | Анкета + check-in + алерт если время на правку > время на ручной отчёт |
| 5 | **Конфиденциальность** | Стратегические данные клиента через Claude API | Anthropic data policy + разделение доступа по ролям |
| 6 | **Over-automation** | AI подменяет суждение коуча шаблонным output | QC-скоринг как информация, не как оценка. Коуч и контроллер решают |
